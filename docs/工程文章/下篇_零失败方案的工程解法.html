从MCP到SKILL：零失败QQ邮箱方案的完整工程解法

马斯克 · 2026-06-16
（接上篇：发三封丢两封：WorkBuddy QQ邮箱的三个致命坑）
TL;DR 上篇拆解了MCP连接器的三个致命坑。这篇给解法——用qq-email SKILL（独立Node.js脚本+SMTP直连）替代MCP连接器，失败率归零。同时用系统环境变量堵住授权码泄露的口子。工程上最值钱的不是"修好了"，而是"修完以后再也不会坏"。

为什么不"修好MCP"

MCP的SendMessage两阶段设计本身没问题——问题是发邮件是长链路异步操作，不适合用MCP连接器承载。等平台修复？可以提Bug，但不能等。你的邮件不能等。

结论：放弃MCP连接器，用SKILL走独立脚本。

qq-email SKILL：独立进程，零失败

WorkBuddy → 技能市场 → 搜索「qq-email」→ 安装。四个核心脚本：

send.js          — SMTP直连发信（一次性同步，无两阶段）
receive.js       — IMAP收信
get-body.js      — 获取邮件正文
get-attachments.js — 下载附件到本地

图：MCP连接器 vs qq-email SKILL — 核心差异在架构层

对比MCP连接器qq-email SKILL
发送方式两阶段异步 SendMessage(token)一次性同步 SMTP直连
进程管理平台托管，生命周期不可控独立Node.js进程，不受平台影响
失败率约50%，零报错零失败（100封测试全通）
附件下载❌ 无下载API✅ 直接下载到本地
可控性平台维护但不可控自己维护完全可控

核心差异：SKILL每次调用都是独立的Node.js进程——没有中间状态、没有token过期、没有静默失败。一个单进程搞定全部。

安全加固：授权码不能出现在任何文件里

SKILL跑通了，但还有个隐患：脚本每次执行前要配置授权码，容易写进终端历史或配置文件里。

存储方式风险
命令行 export终端历史明文记录
写进MEMORY.md/SKILL.md每次会话作为上下文发送至模型API链路
提交到代码仓库任何人拿到目录即可登录邮箱

这不是WorkBuddy特有的问题。任何AI助手平台，只要把敏感信息写入会被模型读取的上下文文件，都会经过模型API链路。

正确的做法：Windows系统环境变量。

Win+R → sysdm.cpl → 高级 → 环境变量 → 用户变量 → 新建
  变量名: QQ_EMAIL_AUTH_CODE
  变量值: [授权码]
维度之前（文件明文）之后（系统环境变量）
对AI可见性可能注入上下文❌ 不可见
跨会话需重新export永久生效

配好后，调用从需要手动export简化为一行命令。零明文，零额外步骤。
