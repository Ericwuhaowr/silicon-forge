## 一、开场：连上了，但不好用

WorkBuddy 左侧连接器面板里有个「QQ邮箱」。点一下，手机扫码，授权通过，状态变绿——`qq-mail: connected`。

没报错。没 500。没超时提示。悄无声息地吞掉了你的邮件。三天后收件人来问："那封邮件呢？"

50% 的失败率，零报错。这是我们用 WorkBuddy QQ邮箱 MCP 连接器踩的第一个，也是最坑的一个。

如果你也是 WorkBuddy 用户，打算用 QQ 邮箱做自动化，这篇文章就是为你写的。

## 二、踩坑实录

### 坑 1：MCP SendMessage 两阶段确认，第二阶段静默失败

| 维度 | 说明 |
|------|------|
| **现象** | `SendMessage` 第一阶段（组装邮件）成功，返回 `confirmation_token`。用户确认后再调用，第二阶段无报错但邮件没发出去 |
| **频率** | 约 50% 失败率。不是每次，而是随机 |
| **根因** | 疑似 MCP 连接器在两阶段之间连接状态丢失（可能是进程回收或心跳超时），第二阶段 token 调用时连接已 `unavailable` |

最坑的是——**它不报错**。没有 500，没有超时提示，悄无声息地吞掉了你的邮件。你发完继续干活，三天后收件人来问"那封邮件呢？"

> **我们怎么发现的？** 连续3次出现"显示发送成功但对方没收到"后，我们写了个脚本循环发100封测试邮件，才复现出这个约50%概率的随机失败。不是每次都挂，但挂的时候没有任何迹象——这也是它最危险的地方。

### 坑 2：附件下载能力缺失

`GetMessage` 能拿到邮件正文，但 `ListAttachments` 之后**没有对应的下载 API**。附件只能看到文件名和大小，取不出来。

这个问题在 100 张发票端到端测试时致命——脚本能搜到发票邮件，能列出附件，就是没法下载 PDF 来提取数据。

### 坑 3：连接器频繁 unavailable

| 症状 | 表现 |
|------|------|
| `GetMe` 返回 `unavailable after recovery` | 连接器显示绿色但功能不可用 |
| 需重新授权 | 关闭对话窗口 → 删除 `.credentials.json` → 重新扫码 |

这不是偶发。从 6 月 10 日到 12 日之间，我们至少经历了 4 次连接器不可用，每次都中断了邮件工作流。

> **我们怎么发现的？** 排查了两天才确认：不是授权过期，是连接器进程被平台回收后，状态显示还是绿色，但实际已经不可用了。`GetMe` 返回 `unavailable after recovery`，唯一的修复方式是关窗口→删凭证→重新扫码。

## 三、解决方案：放弃 MCP 连接器，上 SKILL

### 3.1 为什么不是"修好 MCP"

MCP 连接器的两阶段设计本身没问题，但它是平台侧的问题，我们改不了。等 WorkBuddy 团队修复？可以提 bug report，但不能等。

**结论：绕过 MCP，直接走 IMAP/SMTP。**

### 3.2 qq-email SKILL 的架构

```
用户自然语言 → AI 助手 → qq-email SKILL
                              ├── scripts/send.js     (SMTP 发信)
                              ├── scripts/receive.js    (IMAP 收信)
                              ├── scripts/get-body.js   (获取邮件正文)
                              └── scripts/get-attachments.js (下载附件)
```

| 对比 | MCP 连接器 | qq-email SKILL |
|------|----------|---------------|
| 底层协议 | 平台托管 | Node.js + nodemailer/imap/mailparser |
| 发送方式 | 两阶段确认 | 一次性 SMTP 直连 |
| 失败率 | 约50% | 目前测试零失败（执行链路单一，无状态丢失） |
| 附件下载 | 不支持 | `get-attachments.js` 直接下载到本地 |
| 稳定性 | 依赖平台进程 | 独立进程，不受连接器生命周期影响 |
| 维护成本 | 平台维护但不可控 | 自己维护但完全可控 |

**核心差异：SKILL 是独立脚本，不依赖平台连接器的生命周期。** 每次调用都是一个干净的 Node.js 进程，没有中间状态，没有 token 过期，没有静默失败。

### 3.3 怎么获取 qq-email SKILL

打开 WorkBuddy → 技能市场 → 搜索「qq-email」→ 安装。或者直接在对话中输入「安装 qq-email skill」，AI 助手会自动帮你安装。

安装完成后，SKILL 位于 `~/.workbuddy/skills/qq-email/`，包含四个核心脚本：
- `scripts/send.js` — SMTP 发信
- `scripts/receive.js` — IMAP 收信
- `scripts/get-body.js` — 获取邮件正文
- `scripts/get-attachments.js` — 下载附件到本地

## 四、安全加固：明文授权码的隐患

### 4.1 问题：授权码写在哪

SKILL 跑通了，但脚本每次执行前要 `export` 环境变量，授权码出现在命令行里，会被写入终端历史。

更需要注意的用法是——如果把凭证写进 `MEMORY.md` 或 SKILL.md 等会被 AI 读取的配置文件，这个文件在每次会话时作为上下文注入到 AI 模型中，意味着授权码经过了大模型 API 链路。

> ⚠️ **这不是 WorkBuddy 特有的问题。** 任何 AI 助手平台，只要把敏感信息写入会被模型读取的上下文文件，都会经过模型 API 链路。关键是要有正确的安全意识和做法。

> ⚠️ **不同邮箱的风险等级不同，不能一概而论：**
>
> | 邮箱类型 | 风险等级 | 后果 |
> |---------|:--:|------|
> | 普通个人邮箱 | 🟡 中 | 邮件内容隐私泄露 |
> | 工作邮箱 | 🔴 高 | 商业机密、合同文件暴露 |
> | 绑定了支付/金融的邮箱 | 🔴 极高 | 密码重置→资金风险 |

**这篇文章的核心价值不在"修好了邮路"，而在"修好邮路的同时，意识到并堵住了凭证泄露的口子"。** 如果你只在技术层面解决了问题，却把授权码留在了上下文文件中，那不叫修好，叫换了一种方式埋雷。

| 存储方式 | 风险 |
|---------|------|
| 命令行 `export` | 终端历史明文记录 |
| 写进 MEMORY.md 等配置文件 | 作为上下文发送至模型 API 链路 |
| 提交到代码仓库 | 任何人拿到目录即可登录邮箱 |

### 4.2 解决方案：系统环境变量

```
Win+R → sysdm.cpl → 高级 → 环境变量 → 用户变量 → 新建
  变量名: QQ_EMAIL_AUTH_CODE
  变量值: [你的授权码]
```

| 维度 | 之前 | 之后 |
|------|------|------|
| 存储位置 | 文件明文 | 系统环境变量 |
| 对 AI 可见性 | 可能注入上下文 | 不可见 |
| 跨会话 | 需重新 export | 永久生效 |
| 泄漏面 | 文件 + 终端历史 | 仅本机 |

qq-email SKILL 的脚本本身就从 `process.env` 读取，环境变量配好后，调用方式从：

```bash
export QQ_EMAIL_ACCOUNT=... && export QQ_EMAIL_AUTH_CODE=... && node send.js ...
```

变成：

```bash
node send.js "to@email.com" "主题" "正文"
```

**零明文，零额外步骤。**

> 💡 **如何验证配置成功？** 配完后在终端跑一行：
> ```bash
> node -e "console.log(process.env.QQ_EMAIL_ACCOUNT || '未配置')"
> ```
> 输出你的邮箱地址就说明成功了。

> 💡 **备选方案**：如果有多套邮箱环境（如测试/正式），可以用项目根目录的 `.env` 文件（配合 `dotenv` 包），方便切换。注意：`.env` 务必加入 `.gitignore`。

> 💡 **关于授权码本身**：QQ邮箱的授权码本身就是最小权限设计——只能用于邮件客户端收发，不能登录网页端、不能改密码。即使授权码意外泄露，风险也是可控的。

## 五、工程总结：三层问题，三层解法

| 层 | 问题 | 解法 | 工程原则 |
|------|------|------|---------|
| **传输层** | MCP 连接器不稳定 | IMAP/SMTP 独立脚本 | 不依赖不可控的中间件 |
| **功能层** | 附件无法下载 | mailparser 本地解析 | 核心能力自己做，不交给平台 |
| **安全层** | 凭证明文泄露 | 系统环境变量 | 凭证不与代码/文档共存 |
