## 一、作者背景与核心判断

**归藏**：开发了多款爆款 Agent Skills，包括 PPT Skill、社交媒体卡片 Skill、Logo Generator Skill、AI Desk Card 等。

**核心判断：Agent 放大能力差距，而非抹平差距**

- 头部用户已默认理解 Agent 的组成要素（文档、规则、memory、loop、MCP、CLI、工具调用、权限、安全沙箱等）
- 普通用户只知道"Agent 能写代码""Agent 可以调用 Skill"
- 结果：K 型分化——目标清晰的人被放大，目标混乱的人也被放大混乱
- **Skill 可以弥合 Agent 使用能力差距**

## 二、归藏开发的 SKILL 及其特点

### 1. PPT Skill
- **功能**：文章/大纲→可演示 HTML PPT
- **流程**：读取→询问需求→选主题/颜色/版式→生成→后验检查→修正→输出
- **设计系统**：10种页面布局、5套主题色、三级字体、网格系统、Hero/Non-hero节奏交替
- **起源**：一次AI分享的副产品→开源Skill→**副产品变主产品**

### 2. 社交媒体卡片 Skill
- **功能**：生成3:4竖版图文卡片，适配小红书/公众号/Twitter
- **设计**：11类内容品类、两套视觉系统、28个版式骨架
- **核心**：3:4是主战场，1秒决定停不停下
- **防Slop**：克制AI生图，优先真实摄影和截图美化

### 3. Logo Generator Skill
- **核心思路**：分层处理——SVG本体+展示场景+交互背景，各用最适合的技术
- **技术分工**：人工沉淀审美→模型理解语义→代码稳定排版→图像模型只处理视觉
- **优势**：可控、可改、可复用

### 4. AI Desk Card（AI桌卡）
- **功能**：Agent接管屏幕边缘物理信息位
- **能力**：固件烧录、Wi-Fi配置、推送、定时任务、Todo、日历、墨水屏刷新
- **意义**：Skill边界扩展到物理环境

### 5. 夜巡录（游戏Demo）
- 完整生产流水线：玩法定义→原型→素材采集→绿幕→视频生成→音乐→Electron打包→GitHub Actions→发布
- 跨程序员/美术/动画/作曲/运维的生产流水线

## 三、核心方法论

### Skill本质定义
> "Skill是把专家经验、工作流、品味和工具调用封装成可分发、可复用、可迭代的Agent能力单元。"

| Prompt | Skill |
|--------|-------|
| 容易复制 | 有版本管理 |
| 难以分发 | 可安装、调用、迭代、传播 |
| 一次性对话 | 可反复调用的工作流 |

### 架构设计原则：中心短，辐射厚（Thin Harness, Fat Skills）

```
skill-name/
├── SKILL.md      # 只放高信号流程和判断
├── scripts/      # 确定性逻辑，Agent调用而非重写
├── references/   # 重文档，按条件读取
├── assets/       # 模板、schema、字体、主题、版式
└── 配置文件/      # 首次配置、偏好、历史
```

> "复杂Skill的复杂内容不怕多，怕的是一次性塞给模型。文件系统本身就是一种上下文工程。"

### Description设计：路由触发器，不是宣传语

- ✅ 好："当用户需要把文章、大纲或演讲内容转成可演示HTML PPT时加载"
- ❌ 坏："这个Skill可以生成漂亮PPT"

> 好的描述是Agent的判断条件，坏的是广告。

### 设计类Skill的核心：把品味变成约束

**模型默认收敛到平庸** → 需要注入9条约束规则：
1. 不使用纯白和纯黑
2. 不让用户任意输入hex，只提供验证过的主题色板
3. 不用紫色多彩渐变、发光和大面积blur
4. 动画只用transform和opacity，且在必要时
5. 图文卡片优先真实摄影，AI生图只是兜底
6. 版式骨架人工验证，AI负责填充组合微调
7. 文字自动适配图像主体落点、字色、遮罩、断行
8. 连续三页相同节奏是P0错误
9. 文字不能压脸

> "把AI的任务从'自由设计'降级成'在高质量骨架里填充'。"

### Skill生命周期方法

**6步质量维护法**：
1. 先用无Skill的Agent跑真实任务，找到错在哪里
2. 基于真实query写eval（正例+反例+forbidden load）
3. 先调description，确保该加载时加载、不该加载时不加载
4. 写主体时删显而易见的内容，只保留会改变模型行为的判断
5. 把失败案例追加到gotchas，不不断加长主流程
6. 做跨模型测试

**8步从0到1法**：
发现需求→做一次高质量产物→抽象流程→工程化模板→跨模型测试→封装发布→内容分发→收集反馈

### 成本意识

> "每个Skill都是一种税。进入索引后每个会话都在为name和description付上下文成本；加载后后续对话都在付主体成本。每一句都要问：没有这句，Agent会不会做错？不会就删。"

### gotchas是最高价值内容

> "正向原则往往模型已经知道，负面边界才是专家经验。"

## 四、设计局限与自动进化

| 维度 | Skill | Gene/Capsule |
|------|-------|-------------|
| 来源 | 人类预先沉淀 | 运行中自动长出 |
| 特征 | 明确创建者、边界、流程、版本 | 带成功率、变异历史、适用上下文 |
| 当前可落地性 | 更高 | 长期方向 |

> "不是完全自动，也不是完全手写，而是人定义品味和边界，Agent收集证据、提出改动、补充eval和维护长尾经验。"

## 五、生态与分发

### 分发平台策略
- **GitHub**：基础分发、跨平台覆盖
- **小红书**：视觉内容和使用案例分发
- **应用商店式**：精准推荐、低门槛、可能分成

### Skill展示不应做成仓库列表
每个Skill应展示：解决什么问题、适合什么场景、输出长什么样、典型提示词、失败情况、用户评价、安装方式。

### 防盗策略
不开源也防不住→开源覆盖更多平台→用影响力威慑→持续迭代领先→社区案例形成品牌资产

### 内容复利飞轮
> "Skill→文章→案例→开源仓库→社交反馈会互相喂养。这是个人产品在Agent时代的复利飞轮。"
