# 主对话窗口效率优化方案 V1.0

> 基于 2026-06-16 全天会话模式分析 · 吴昊 + 马斯克 对齐定稿
> 核心原则：**高质量完成任务是首要目标，算力节约服从质量底线**

---

## 〇、核心原则：质量门控

任何优化方案在采纳前，必须先回答一个问题：

> **"这个优化是否有可能让任务结果变差？"**

| 答案 | 处理 |
|:--:|------|
| 零可能 | ✅ 直接采纳 |
| 有可能但不致命 | ⚠️ 加安全网后采纳 |
| 有可能且不可接受 | ❌ 砍掉 |

**吴昊原话**："如果节约了算力但影响了任务完成质量，其实是更大的浪费——因为除了算力，我们还消耗了时间、精力等更稀缺的资源。"

---

## 一、当前问题诊断

基于 2026-06-16 会话的实际 tool call 模式分析：

| 问题 | 严重度 | 具体表现 | 浪费类型 |
|------|:--:|------|:--:|
| **邮件串行处理** | 🔴 | 每次"收邮件→读→分析→回"独立循环，每天~10次，每次5-8 tool calls | 结构性浪费 |
| **纯确认也回复** | 🟡 | 德米斯的"收到了""确认了"类回复也触发完整回信 | 决策性浪费 |
| **工具调用串行** | 🟡 | 多封邮件/多张图片本可并行读取却串行 | 时序性浪费 |
| **记忆碎片化写入** | 🟢 | 每完成一个子任务就 Edit 一次 memory | 习惯性浪费 |

语境说明：我们的主对话窗口承载复杂、高专业度的任务（公众号文章全流程生产、SKILL方法论开发、三方协作），对上下文精确性和完整性要求极高。

---

## 二、质量门控后的保留方案（4项）

以下方案已逐一通过质量门控审查，确认对任务质量零负面影响。

### ① 邮件并行读取（P0，立即启用）

**现状**：
```
receive.js → get-body.js(UID 579) → 分析 → 回复
receive.js → get-body.js(UID 580) → 分析 → 回复  ← receive 重复调用
receive.js → get-body.js(UID 581) → 分析 → 回复  ← receive 再次调用
```

**优化**：
```
receive.js（一次）→ [get-body.js(UID 579) + get-body.js(UID 580) + get-body.js(UID 581)]（并行）
→ 批量分析 → 逐一回复（或标记需回复的）
```

| 维度 | 评估 |
|------|:--:|
| 质量影响 | ✅ 零 — 每封邮件仍然完整读取，信息完整度不变 |
| 效率提升 | receive.js 从 N 次 → 1 次，get-body 从串行 → 并行 |
| 预计节省 | 邮件处理环节 50%+ tool calls |

### ② 用户判读是否回复（P0，立即启用）

**现状**：用户说"收哈萨比斯邮件，需要回复请回复"，默认每封都回。

**优化**：收件后只给摘要，标注"是否需要回复？"——由吴昊判断。

| 场景 | 处理 |
|------|------|
| 德米斯纯确认（"收到""完全同意"） | 提示：无需回复 |
| 德米斯有新决策/问题 | 提示：建议回复（给出要点） |
| 德米斯发来新产出（合订版等） | 提示：建议回复并审阅 |

| 维度 | 评估 |
|------|:--:|
| 质量影响 | ✅ 零 — 回复决策权100%在吴昊手里，不会漏回重要邮件 |
| 效率提升 | 免去"纯确认也回一封"的浪费（昨天至少3-4封） |
| 风险控制 | 如果吴昊判断应回但我们没标注，误漏概率极低（摘要已包含全部信息） |

### ③ 独立操作并行化（P0，立即启用）

**现状**：多张图片/多个文件逐个串行Read。

**优化**：无依赖关系的独立操作同步发出。

典型案例：

| 场景 | 串行（旧） | 并行（新） |
|------|------|------|
| 多张LOVART封面分析 | Read图1 → Read图2 → Read图3 | Read图1+2+3同时 |
| 多封德米斯邮件 | get-body逐个 | get-body批量并行 |
| 多插图截图 | playwright逐个open | playwright批量open |

| 维度 | 评估 |
|------|:--:|
| 质量影响 | ✅ 零 — 并行操作互相独立，结果和串行完全一致 |
| 效率提升 | 3个独立Read节省2/3等待时间 |
| 适用条件 | 仅限无依赖关系的操作；前后依赖的必须串行 |

### ④ 记忆合并写入（P1，本周启用）

**现状**：
```
完成子任务A → Edit memory.md  ← 碎片
完成子任务B → Edit memory.md  ← 碎片
完成子任务C → Edit memory.md  ← 碎片
```

**优化**：
```
完成完整阶段（如"封面方法论对齐完毕"）→ Edit memory.md（一次性追加全部条目）
```

| 维度 | 评估 |
|------|:--:|
| 质量影响 | ✅ 零 — 写入内容总量不变，只是合并时机 |
| 效率提升 | 减少碎片化 Edit 调用 |
| 适用条件 | 一个自然工作阶段结束（如邮件对齐完毕、文章定稿、方法论合订完成） |

## 三、明确砍掉的方案（质量风险不可接受）

| 原方案 | 砍掉原因 | 具体风险 |
|------|------|------|
| 跳过验证Read | 工具返回成功≠文件内容正确。写过垃圾代码才知道文件可能为空、截断、编码错。 | 用户看到的文件可能是坏的 |
| 邮件内容摘要化 | 德米斯邮件常有嵌套的技术判断和精确表述。摘要→丢失细节→回复偏差→沟通成本反而更高。 | 少读两个字，多回三轮邮件 |
| Agent子代理卸机械 | 子代理没有本会话的完整上下文（三方协作框架、前文决策、用户偏好），脱离上下文等于盲干。 | 子代理的产出可能与主会话脱节 |
| 上下文门控 | 选择性加载记忆可能导致"该用的没加载"，我们的任务对上下文完整性要求极高。 | 关键记忆遗漏导致判断偏差 |
| Workspace拆分 | 目前所有工作流高度交织（封面→插图→写作→方法论的产物互相引用），拆分会增加跨空间引用成本。 | 拆了反而更复杂 |

## 四、已完成的优化（飞书群替代邮件）

2026-06-16 23:49，三方协作群「AI内容工厂 · 三方协作群」在飞书建立，德米斯已通过吴昊账号授权接入。

| 操作 | 旧（邮件） | 新（飞书群） |
|------|------|------|
| 日常同步 | 收→读→回，5-8 steps | 群里一条消息 |
| 快速决策 | 异步邮件，等待不定 | 群内实时 |
| 文件附件 | 邮件附带，对方可能打不开 | 飞书云盘/文档共享链接 |
| 会议记录 | 无 | 群聊天然沉淀，抄送后续 |

**约定**：飞书群负责日常同步、进度讨论、临时碰撞。邮件保留用于正式交付/归档/需要附件记录的场合。
