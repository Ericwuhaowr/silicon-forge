# WorkBuddy 基础设置优化指南 — 从零到高级的三阶路径（完整版）

> 2026-06-18 | 基于SKILL巡检 + 飞书邮件三方协同 + 基础设施完善三个任务的实战提炼
> **完整版**：融合WB侧实战经验 + 扣子侧视角补充（4个新增维度 + 分级微调 + 3项实践）
> 面向 WorkBuddy 用户，按初级/中级/高级三级分层

---

## 为什么要做基础设置优化？

WorkBuddy 默认配置能完成日常任务，但面对**多轮复杂任务、多智能体协作、持续迭代**等高阶场景时，"裸机配置"会暴露三个瓶颈：

| 瓶颈 | 表现 | 后果 |
|------|------|------|
| 记忆衰减 | 跨会话后之前的决策/偏好/规则丢失 | 每次对话从零开始，重复解释、重复踩坑 |
| 协作摩擦 | 多人/多AI协作没有统一规则和通信协议 | 信息差、重复劳动、决策延迟 |
| 资产散落 | 产出物散落各目录，没有统一索引和归档标准 | 找资料的成本高于写资料，知识不可复用 |

基础设置优化的核心目标：**把"一次性消耗"升级为"可复利增长的资产"**。

---

## 三级路径总览

| 级别 | 适合谁 | 投入时间 | 核心能力 |
|:--:|------|:--:|------|
| **初级** | 刚接触WorkBuddy，日常独立使用 | 30分钟 | 记忆初始化 + 日志建立 |
| **中级** | 有多轮复杂任务需求，开始协作 | 2小时 | 规则分层 + 文档标准化 + 协作协议 |
| **高级** | 跨平台多智能体协作，持续方法论迭代 | 持续优化 | 三层记忆架构 + 方法论资产化 + 自进化机制 |

## 一、初级用户：30分钟建立"记忆基线"

### 适用场景

刚安装 WorkBuddy，日常用于：写作、查询、简单编码、邮件处理等独立任务。

### 现状痛点

- 每次新会话，AI 不知道你的偏好、工具配置、常用路径
- 踩过的坑下一次还会踩——因为没有记录下来

### 方法与技术

#### 1.1 初始化 MEMORY.md（10分钟）

在项目目录 `.workbuddy/memory/` 下创建 `MEMORY.md`，写入以下基础内容：

```
# 项目记忆

## 一、我的偏好
- 回复风格：[简洁/详细]，偏好[表格/文字]格式
- 时间表述：精确到日期和时间（如2026-06-18 16:00），不用"明天""下周"

## 二、工具配置
- 编辑器：VS Code（路径：C:\Users\xxx\...）
- Node.js版本：22.x
- 包管理器：pnpm

## 三、常用服务
- QQ邮箱：账号 xxx@qq.com，通过环境变量 QQ_EMAIL_ACCOUNT 和 QQ_EMAIL_AUTH_CODE 使用
- GitHub用户名：xxx
```

**达到效果**：新会话自动加载这些信息，AI 不再问"你用哪个邮箱？""你偏好什么格式？"

#### 1.2 建立每日工作日志（5分钟）

在 `.workbuddy/memory/` 下创建 `YYYY-MM-DD.md`，每完成一个子任务追加一行：

```
## 任务名称 — 14:00-14:30
- 做了什么：[一句话描述]
- 产出物：[文件路径]
- 使用的SKILL：[如有]
```

**达到效果**：下次会话 AI 读取日志，知道你昨天做到哪了、产出了什么、有什么遗留问题。

#### 1.3 安装核心 SKILL（15分钟）

推荐安装以下三个基础 SKILL：

| SKILL | 作用 | 安装命令 |
|------|------|------|
| `quality-gate` | 交付前自动质量检查 | 搜索"质量门禁"安装 |
| `review-checklist` | 结构化审阅清单 | 搜索"审阅清单"安装 |
| `qq-email` / 对应邮箱 SKILL | 邮件收发自动化 | 按实际邮箱安装 |

**达到效果**：AI 会自动在交付前检查事实准确性、格式一致性，不会出现低级错误。

#### 1.4 编写系统提示词基础配置（10分钟）※ v2新增

系统提示词是 AI 的"默认行为说明书"。很多用户忽略了这一步，导致 AI 输出风格、格式、深度都不稳定。

**操作步骤**：
1. 在 `~/.workbuddy/MEMORY.md`（用户级，跨项目生效）中写入偏好
2. 核心三要素必须定义清楚：

| 要素 | 示例 | 为什么重要 |
|------|------|------|
| 回复风格 | "简洁，结论先行，表格优先" | 避免AI唠叨或废话连篇 |
| 表达规范 | "时间精确到日期+时间，不用'明天''下周'" | 消除模糊指令带来的执行偏差 |
| 错误处理 | "不确定的标注'待确认'，禁止虚构数据" | 守住真实性底线 |

**达到效果**：AI 行为一致性大幅提升，不再每次需要重新"调教"回答风格。

## 二、中级用户：2小时建立"协作规则层"

### 适用场景

任务复杂度升级：多轮协作、跨平台沟通、定期巡检、多次迭代的文档/代码。

### 现状痛点

- MEMORY.md 越写越长，查找信息变慢
- 和协作者（人或AI）之间信息不同步，决策延迟
- 文件散落各处，找不到上一次的产出物

### 方法与技术

#### 2.1 MEMORY.md 分级写入（Q1/Q2/Q3）

不再把所有东西塞进一个文件。按重要程度分级：

**分级标准（来自MEMORY.md分级框架）**：

| Q | 问自己 | 存哪里 | 示例 |
|:--:|------|------|------|
| **Q1** | "忘了这个下次会出错吗？" | MEMORY.md 正文 | 质量红线、安全规则、双盲流程 |
| **Q2** | "下次任务需要主动想起吗？" | MEMORY.md 摘要+指针 | 重要决策和经验教训 |
| **Q3** | "只是未来可能需要查询吗？" | 独立文档+文件名索引 | 历史报告、原始数据 |

**操作步骤**：
1. 扫描现有 MEMORY.md，用三个问题对每条信息分级
2. Q3 内容移出到独立文档，MEMORY.md 只留一行指针（如 `详见 docs/方案/xxx.md`）
3. Q1 内容放在 MEMORY.md 开头（自动加载最先生效）
4. 新会话启动后检查 Q1 是否完整

**达到效果**：MEMORY.md 从"杂货铺"升级为"速查手册"。核心规则永远最先加载，详细信息不挤占上下文窗口。

#### 2.2 建立文件命名与目录规范

**命名模板**：`{主题}-{版本}-{日期}.{扩展名}`

| 示例 | 含义 |
|------|------|
| `巡检报告_第3期_v1.0_2026-06-18.md` | 第3期巡检v1.0，2026年6月18日 |
| `协作流程_联合版_v1.1_2026-06-18.md` | 联合版v1.1，同日 |

**目录结构模板**：

```
项目根目录/
├── docs/
│   ├── 方法论&文档/     ← 可复用的方法论、流程、模板
│   ├── 项目输出/        ← 按项目归档的产出物
│   └── 外部参考/        ← 下载的外部资料、研究笔记
├── .workbuddy/
│   └── memory/          ← Q1规则层 + 每日日志
└── README.md            ← 项目总索引
```

**达到效果**：任何文档5秒内定位。新协作者通过 README 快速了解全局。

#### 2.3 建立双渠道协作模型

如果你有协作者（人或AI），建立"实时+归档"双渠道：

| 渠道 | 用途 | 不用于 |
|------|------|------|
| **即时通讯**（飞书/微信/企业微信） | 日常同步、快速决策、进度讨论 | 正式交付、需要归档的内容 |
| **邮件** | 正式报告、里程碑决策、附件交付 | 日常讨论、即时同步 |

**核心规则**：
- 日常讨论走即时通讯，邮件只用于正式交付
- 重要决策必须在邮件留档一份，确保可追溯
- 纯确认（"收到""好的"）不触发邮件

**达到效果**：沟通延迟从小时级降到分钟级，正式产出永不丢失。

#### 2.4 引入消息分级处理

面对协作者的消息/邮件，建立三级自动判断：

| 级别 | 定义 | 例子 | 动作 |
|:--:|------|------|------|
| 🟢 自主处理 | 信息同步、技术分享 | 日报、进度更新 | AI 阅读后摘要汇报 |
| 🟡 需确认 | 需要你拍板的决策 | 方案选择、优先级调整 | AI 收集选项后提交你确认 |
| 🔴 必须等决策 | 涉及权限/费用/对外 | 授权、预算、合同条款 | AI 绝不自行处理，标记等你 |

**达到效果**：70%+ 的日常消息自动处理，重要事项 100% 上报。不被信息洪流淹没，也不漏掉关键决策。

#### 2.5 核心SKILL选型清单（30分钟）※ v2新增

WorkBuddy 有丰富的 SKILL 市场，但"装得多不如装得对"。按场景推荐选型：

| 场景 | 推荐SKILL | 作用 | 优先级 |
|------|------|------|:--:|
| 质量保障 | `quality-gate` + `review-checklist` | 交付前铁律检查 + 结构化审阅 | 🔴 必装 |
| 邮件处理 | `qq-email` / 对应邮箱SKILL | IMAP收件+SMTP发件自动化 | 🔴 必装 |
| 写作输出 | `wuhao-writer-*` | 公众号科普文/工程文章的写作规范 | 🟡 推荐 |
| 金融分析 | `wb-finance-skill` / `westock-data` | 财报分析/估值/选股/盘后数据 | 🟡 按需 |
| 合同审查 | `contract-review-pro` | 双栏对照版审查报告 | 🟡 按需 |
| 内容创作 | `content-creator` / `knowledge-card-factory` | 小红书图文/知识卡片/信息图 | 🟢 按需 |
| 多格式交付 | `multi-format-delivery` | MD→DOCX/PPT/知识卡片批量转换 | 🟢 按需 |

**达到效果**：避免"装了20个SKILL但常用的只有3个"的浪费。按场景精准选型，每个SKILL都有明确的调用时机。

#### 2.6 凭证与密钥安全管理（15分钟）※ v2新增

多服务接入时，凭证管理是最容易被忽视的安全隐患。

**核心规则**：

| 规则 | 做法 | 后果如果违反 |
|------|------|------|
| 不落地 | API Key/Token 写入环境变量，绝不出现在代码中 | 代码泄露=凭证泄露 |
| 最小权限 | 每个连接器只申请必需的最小权限范围 | 权限泛滥=攻击面扩大 |
| 分级授权 | 开发/测试/生产用不同凭证 | 测试失误不影响生产 |
| 定期轮换 | 每季度更换一次授权码和Token | 降低长期泄露风险 |

**操作示例（QQ邮箱）**：
```
# Windows 系统环境变量
QQ_EMAIL_ACCOUNT=xxx@qq.com
QQ_EMAIL_AUTH_CODE=授权码（非QQ密码）

# 代码中只引用变量名，不写真实值
const account = process.env.QQ_EMAIL_ACCOUNT
```

**达到效果**：核心服务凭证零泄露风险，即使某次会话日志被外传，也不会暴露敏感信息。

## 三、高级用户：建立"自进化方法论体系"

### 适用场景

跨平台多智能体协作、持续迭代的方法论开发、SKILL 生态建设。

### 现状痛点

- 每次任务产出是一次性消耗，知识无法复用
- 协作模式凭经验判断，没有标准化流程和量化指标
- 多个智能体之间没有统一的"作战手册"

### 方法与技术

#### 3.1 三层记忆架构

从单层 MEMORY.md 升级为三层体系：

| 层级 | 存放内容 | 加载时机 | 示例 |
|:--:|------|:--:|------|
| **Q1 即时层** | 核心规则 + 当前状态 | 新会话自动注入 | MEMORY.md |
| **Q2 近中期层** | 项目进度、决策记录、方法论 | 按需加载 | `docs/方法论&文档/` |
| **Q3 长期层** | 历史报告、原始数据 | 检索式访问 | `docs/项目输出/` 归档 |

**同步策略**：
- Q1：事件驱动同步——任何一方修改 Q1 核心规则，立即通知另一方
- Q2：节点驱动同步——每个项目里程碑完成后，双方对表更新
- Q3：按需同步——需要时才检索，不主动推送

**达到效果**：核心信息秒级访问，详细信息不丢失，记忆容量无限扩展。团队规模扩大不会导致信息管理崩溃。

#### 3.2 双盲协作模式

对于高价值任务（质量要求高、影响深远），采用"先独立、再对齐"模式：

| 阶段 | 做法 | 解决的问题 |
|:--:|------|------|
| ① 独立分析 | 双方各从自己视角独立出完整方案 | 避免锚定效应——先入为主 |
| ② 交叉对齐 | 对比差异点，讨论后统一 | 差异点暴露盲区，互相补充 |
| ③ 联合交付 | 融合双方最优部分 | 结果 > 任何一方单独完成 |

**终止条件**：僵局时→换位复述确认理解→提交决策者仲裁→如仍不确定则AB测试并行验证。从僵局到触发仲裁不超过30分钟。

**达到效果**：结论可信度翻倍（两个独立思维从不同路径走到同一结论）。盲区概率大幅降低。

#### 3.3 趁热打铁闭环

任务启动后必须在"上下文热度周期"内闭环。时间窗口从任务中断开始计算：

| 规模 | 窗口 | 超时处理 |
|:--:|:--:|------|
| 小（单步骤） | 1小时内 | 冷启动，需重新对齐 |
| 中（多步骤） | 4小时内 | 简要回顾上下文后继续 |
| 大（跨天） | 当日闭环 | 状态快照同步双方 |

**达到效果**：消除"等明天再说"带来的指数级重入成本。任务连贯性从50%提升到90%+。

#### 3.4 方法论资产化

每次任务闭环后，完成五个步骤的复盘沉淀：

```
成果总结 → 经验提炼 → 方法论沉淀 → 工具化落地 → 知识库更新
```

**复盘三问**：
1. 做对了什么？→ 固化到流程/规则
2. 哪里能更好？→ 列出下期改进项
3. 什么可推广？→ 扩大适用范围

**工具化落地**：将有效做法固化为 SKILL、规则层（MEMORY.md）、自动化日程、标准模板。

**达到效果**：每次任务都在为下次任务"铺路"。三个月后，80%的新任务可以直接复用已有方法论，不再从零开始。

#### 3.5 质量保障三层验证

```
第一层：自我校验 → 执行方自己检查（交付前）
第二层：交叉验证 → 另一方独立评审（交付后）
第三层：终审确认 → 决策者最终把关（发布前）
```

搭配**质量门控原则**：任何优化措施必须先回答"这会不会让结果变差？"
- 不影响质量 → 直接推进
- 轻微影响但大幅提效 → 加安全网，需确认
- 牺牲质量 → 绝对不做

**达到效果**：交付物质量从"凭感觉"升级为"有机制保障"。关键错误率趋近于零。

#### 3.6 工作流编排与自动化（持续）※ v2新增

当任务复杂度超过单步 SKILL 能处理的范围时，需要将多个 SKILL 串联为工作流。

**典型编排场景**：

| 场景 | 串联链路 | 说明 |
|------|------|------|
| 巡检→报告→发布 | SKILL巡检 → 质量门控 → 邮件发送 | 巡检完成后自动检查质量，通过后邮件发布 |
| 写作→配图→发布 | 写作SKILL → 配图SKILL → 格式转换 → 发布 | 一篇文章从创作到发布的全链路 |
| 数据分析→洞察→报告 | 数据查询 → 分析框架 → 报告生成 → 邮件推送 | 定期数据报告自动化 |

**编排原则**：
- 每个节点都有明确的输入/输出格式
- 失败节点不阻塞后续（除非有依赖关系）
- 关键节点后加质量门控检查点

**达到效果**：把"手动点5次"的操作压缩为"一句话触发"。复杂任务从"每次做30分钟"降到"触发后等结果"。

#### 3.7 记忆代谢机制（月度）※ v2新增

不是所有规则都值得永远保留。过时的、失效的经验会拖慢检索速度、占用上下文窗口、甚至误导新决策。

**代谢操作流程**：

| 频率 | 动作 | 目标 |
|:--:|------|------|
| 每月 | 扫描 MEMORY.md + 日志文件 | 标记3个月以上未使用的规则为"待审查" |
| 每季度 | 清理过时规则 | 删除已不再适用的经验、已完成的临时项目记录 |
| 每次任务闭环 | 检查 Q2 指针 | 确认引用的文件仍然存在、内容仍然有效 |

**代谢标准**：
- 项目已结束 > 30天 → Q2降级到Q3（移出主动加载区）
- 规则连续3个月未触发 → 标记废弃 → 下次审计确认删除
- SKILL 已卸载 → 相关配置同步清理

**达到效果**：MEMORY.md 保持在300行内，Q2指针控制在15条以内。记忆不臃肿，每次加载的都是"活"信息。

#### 3.8 Prompt资产化管理（持续）※ v2新增

高效的 Prompt 不是一次性的指令，而是可以版本化管理的资产。

**资产化管理要素**：

| 要素 | 做法 | 示例 |
|------|------|------|
| 版本化 | 重要 Prompt 建立版本号 | `任务拆解模板_v2.1_2026-06-18` |
| 场景标签 | 按适用场景打标签 | #代码审查 #写作 #数据分析 |
| 效果追踪 | 记录Prompt的实际产出质量 | "v1.0准确率70%，v1.1加了示例后提升到90%" |
| A/B测试 | 同一任务用不同Prompt并行 | 两版Prompt各跑一遍，选优固化 |

## 五、跨平台补充实践（扣子侧贡献）※ v2新增

以下实践来自扣子侧的独立经验，可与 WB 侧方法论互补使用：

### 5.1 质量门控落地方法

| 环节 | 做什么 | 示例 |
|------|------|------|
| 前置检查 | 任务开始前：输入完整吗？目标清晰吗？ | 巡检前检查源列表是否覆盖所有目标平台 |
| 后置校验 | 交付前：事实准确吗？逻辑自洽吗？ | 报告生成后用 quality-gate SKILL 逐条验证 |
| 阈值设定 | 定义"通过"的标准 | 巡检评级差异 > 2级 → 触发人工复审 |

### 5.2 话题追踪标准化流程

用于复杂项目的上下文管理：

```
阶段1: Setup Token → 为项目创建唯一标识，关联所有相关会话
阶段2: Briefing   → 每个新会话开始时，用Token提取上下文摘要
阶段3: Verify-Run → 确认提取的上下文完整后，再执行任务
```

**适用场景**：持续多天的项目、跨会话的协同任务。

### 5.3 跨平台协作接口规范

当两个平台（如WB + 扣子）协作时，需要约定：

| 维度 | 约定 |
|------|------|
| 文件格式 | 双方统一使用 Markdown（.md），确保跨平台可读 |
| 同步频率 | Q1规则：事件驱动（变更立即通知）/ Q2文档：节点驱动（里程碑后同步）/ Q3档案：按需检索 |
| 版本管理 | 语义化版本号（v主.次），大版本（v2.0）需三方评审 |
