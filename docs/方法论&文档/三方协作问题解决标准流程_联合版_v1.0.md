## 一、核心理念

### 1.1 碳基瓶颈 vs 硅基突破

人类协作存在三大约束，在三方模式下被系统性突破：

| 约束类型 | 碳基瓶颈 | 硅基解决方案 |
|---------|---------|-------------|
| **高摩擦** | 对齐成本高、沟通损耗大 | 结构化流程 + 飞书/邮件双渠道 + 明确分工 |
| **熵增** | 信息衰减、上下文遗忘、走样 | 记忆分层存储 + 文档化沉淀 + 定期校准 |
| **生理极限** | 疲劳、遗忘、精力有限 | 7×24 持续运转 + 上下文热启动 + 跨会话接力 |

### 1.2 趁热打铁原则

> **在衰减前完成才叫完成。**

重要任务一旦启动，必须在上下文热度周期内闭环。离开任务后重新进入的成本是**指数级**的——不仅是算力消耗，更是认知链路的重新建立成本。

**实践口诀**：对齐 → 趁热 → 闭环 → 总结 → 增强

每完成一次完整闭环，就是在给"集体神经链路"做**髓鞘质强化**——链路更粗、更快、更可靠。

**趁热三要素**：
- 方案确认后立即分配执行（不等"计划好再说"）
- 双端并行互不阻塞（你改你的，我改我的）
- 规则同步不滞后（代码和文档同时更新）

### 1.3 价值主张

我们探索的不只是工具和流程，是**碳硅协同进化**的一种新形态——碳基提供方向感、价值观、审美品味；硅基提供算力、记忆、执行力。两者结合，才能做任何一方单独做不成的事。

### 1.4 适用场景快速判断（v1.1 新增）

**四级场景分级（S/A/B/C）**：

| 级别 | 特征 | 示例 | 是否双盲 |
|:--:|------|------|:--:|
| **S** | 质量要求最高、影响深远、安全关键 | SKILL巡检、重大方案决策、安全审计 | 强制 |
| **A** | 质量要求较高、需多视角碰撞 | 方案设计、问题排查、方法论沉淀 | 推荐 |
| **B** | 中等复杂度、单一视角基本够用 | 常规报告、简单分析 | 可选 |
| **C** | 纯执行、时效性极强 | 发消息、传文件、紧急响应 | 不建议 |

**3问快速判断法**：
1. 这个任务出错会导致什么后果？→ 不可接受 → S级
2. 单一视角能保证质量吗？→ 不能 → A级
3. 是否有时间做双端独立分析？→ 没有 → 降级为串联

## 二、五步法标准流程

### 阶段 0：问题识别 — 发现真问题

**目标**：从现象中剥离出真正需要解决的核心问题，避免用正确方法解决错误问题。

| 步骤 | 动作 | 责任人 | 输出物 |
|:--:|------|--------|--------|
| 0.1 | 现象描述：客观描述发生了什么，不做归因判断 | 发现者 | 现象清单 |
| 0.2 | 影响评估：这个问题造成了什么具体影响（质量/效率/成本/体验） | 发现者 | 影响量化 |
| 0.3 | 根因追问：连续问 3-5 个"为什么"，挖到最底层原因 | 双方独立 | 根因假设 |
| 0.4 | 问题定义：用一句话清晰定义"解决什么问题、达到什么目标" | 交叉对齐 | 问题定义文档 |

**质量门**：问题定义必须同时回答「是什么」和「不是什么」，避免范围蔓延。

**真问题三问检验**：
1. 是真实数据还是主观感受？
2. 影响面多大？（质量/效率/成本/体验）
3. 我们有没有能力解决？

**工具**：使用「问题描述模板」（见第五章 §5.1）。

> **案例**：算力优化 — 吴昊看算力消耗截图 → 发现"查飞书/查邮件"消耗高 → 马斯克独立分析四大根因 → 确认是真问题 → 进入阶段1。

---

### 阶段 1：独立分析 — 互不干扰，各自产出

**目标**：在不受对方思路影响的前提下，各自产出完整的分析框架和解决方案。

| 步骤 | 动作 | 责任人 | 输出物 |
|:--:|------|--------|--------|
| 1.1 | 信息收集：各自独立获取所需资料和数据 | 德米斯 + 马斯克 | 原始素材 |
| 1.2 | 框架搭建：各自建立分析框架和评估维度 | 德米斯 + 马斯克 | 分析框架 |
| 1.3 | 方案设计：各自独立产出完整解决方案 | 德米斯 + 马斯克 | 独立方案 |
| 1.4 | 自我校验：各自按质量标准自检一遍 | 德米斯 + 马斯克 | 自检报告 |

**关键原则**：
- 🚫 **禁止提前交流思路**——锚定效应是双盲模式的头号敌人
- ✅ **鼓励使用不同方法**——方法论差异越大，交叉价值越高
- ✅ **允许结论不同**——差异点正是后续对齐的核心价值所在
- ✅ **方案必须是完整版**——不能是草稿，否则分不清"真实结论"还是"还没想清楚"

**适用范围**：凡是双方都能独立访问的问题域（SKILL巡检、技术评估、方案设计等），默认采用双盲模式。

**工具**：使用「独立分析输出模板」（见第五章 §5.2）。

> **案例**：WB侧从主窗口视角分析（四大根因+三优化）；扣子侧从自动化视角分析（双轨重复+全量冗余+独立session开销）。两个框架完全不同但结论高度一致。

---

### 阶段 2：交叉对齐 — 差异碰撞，共识升级

**目标**：通过对比双端独立方案，发现盲区、消除偏差、形成更优解。

| 步骤 | 动作 | 责任人 | 输出物 |
|:--:|------|--------|--------|
| 2.1 | 方案交换：双端同步各自完整方案 | 德米斯 + 马斯克 | 双方方案 |
| 2.2 | 差异识别：逐条对比，标记相同点和差异点 | 双方各自 | 差异对比表 |
| 2.3 | 深度讨论：针对差异点展开讨论，各自说明依据和逻辑 | 德米斯 + 马斯克 | 讨论纪要 |
| 2.4 | 共识收敛：整合双方最优部分，形成联合方案 | 德米斯 + 马斯克 | 联合方案初稿 |

**差异处理原则**：

| 差异类型 | 处理方式 |
|---------|---------|
| **事实类差异** | 核实原始数据和来源，以更准确的为准 |
| **判断类差异** | 各自陈述逻辑链，看谁的论证更完整、证据更充分 |
| **优先级差异** | 回归问题定义和战略目标，以全局最优为准 |
| **审美/风格类差异** | 不纠结，谁执行谁决定；或双版本并行验证 |

**双渠道模式**：
- 📧 **邮件发肉**：正式报告、完整方案、留档资料 → 邮件，确保可追溯
- 💬 **飞书递筷子**：简要通知、进度同步、轻量讨论 → 飞书，确保触达效率

**双盲终止条件（v1.1新增）**：
当交叉讨论陷入僵局、双方无法达成共识时，按以下步骤收场：
1. **换位复述**：双方各自复述对方的论据和逻辑链，确认完全理解了对方立场
2. **吴昊仲裁**：仍无法共识时提交吴昊做最终裁决
3. **AB测试**：如吴昊也无法确定，双方方案各做一版并行验证，用结果说话
4. **时效**：从僵局识别到触发仲裁，不得超过30分钟——避免无休止讨论

> **高阶心法**：不要试图"说服对方"，而是一起"寻找更优解"。当发现对方和你不一致时，第一反应不应该是"他错了"，而是"他看到了什么我没看到的？"

**工具**：使用「交叉对齐检查清单」（见第五章 §5.3）。

---

### 阶段 3：落地执行 — 分阶段推进，快速验证

**目标**：将联合方案拆解为可执行的步骤，按节奏落地，边做边调。

| 步骤 | 动作 | 责任人 | 输出物 |
|:--:|------|--------|--------|
| 3.1 | 任务拆解：将方案拆分为具体可执行的子任务 | 执行方 | 任务拆解清单 |
| 3.2 | 优先级排序：按影响大小 + 实施难度排序 | 执行方 | 优先级矩阵 |
| 3.3 | 第一阶段落地：先做见效最快、风险最低的部分 | 执行方 | 阶段一成果 |
| 3.4 | 效果验证：快速验证第一阶段效果，确认方向正确 | 双方交叉 | 验证报告 |
| 3.5 | 全量推进：验证通过后，推进剩余阶段 | 执行方 | 完整落地成果 |

**执行四原则**：
1. **趁热打铁**：三方对齐后立即动手，不拖延
2. **双端并行**：互不阻塞，各自动手，效率最大化
3. **分阶段**：大方案拆小步，每步可验证、可回滚
4. **规则同步**：代码和文档同时更新（代码改了，MEMORY.md跟着改）

**趁热打铁时间窗口（v1.1量化）**：
时间从"任务中断"那一刻开始计算，非从"任务开始"计算：

| 任务规模 | 窗口 | 超时处理 |
|:--:|:--:|------|
| 小（单步骤） | 1小时内 | 超过视为冷启动，需重新对齐 |
| 中（多步骤） | 4小时内 | 超过需简要回顾上下文后再继续 |
| 大（跨天） | 当日闭环 | 隔天必须做状态快照同步双方 |

**节奏控制**：
- 第一阶段追求**快速见效**——让大家尽快看到正向反馈
- 每个阶段结束后必须有**明确的验收标准**
- 发现问题立即调整，**不把问题拖延到下一轮**

---

### 阶段 4：复盘沉淀 — 方案 → 可复用方法论

**目标**：从单次成功中提炼可复用的方法论，让每次成功都成为下次的台阶。

| 步骤 | 动作 | 责任人 | 输出物 |
|:--:|------|--------|--------|
| 4.1 | 成果总结：本次任务达成了什么、没达成什么 | 双方联合 | 成果总结 |
| 4.2 | 经验提炼：哪些做法有效、哪些走了弯路、为什么 | 双方联合 | 经验教训清单 |
| 4.3 | 方法论沉淀：将有效做法抽象为可复用的流程、模板、检查清单 | 双方联合 | 方法论文档 |
| 4.4 | 工具化落地：将方法论固化为 SKILL、规则、自动检查机制 | 双方各自 | 落地工具 |
| 4.5 | 知识库更新：将成果和方法论写入记忆体系和资料库 | 双方各自 | 更新后的知识库 |

**复盘快速启动：三问自检**（嵌入 4.1 之前）：
1. 这次做对了什么？ → **固化**到流程/规则中
2. 哪里还能更好？ → **列出**下期改进项
3. 什么可以推广到其他任务？ → **扩大**适用范围

**沉淀标准**：
- 🎯 **可复用**：下次遇到同类问题，拿起来就能用，不需要重新想
- 📐 **可衡量**：有明确的判断标准和验收指标，不是凭感觉
- 🔄 **可迭代**：留有升级空间，下次实践后可以优化更新

**工具**：使用「闭环复盘模板」（见第五章 §5.4）。

## 三、质量保障体系

### 3.1 三层验证机制

```
第一层：自我校验 → 执行方自己检查（阶段1.4）
第二层：交叉验证 → 另一方独立评审（阶段2）
第三层：终审确认 → 吴昊最终把关（贯穿全程）
```

### 3.2 质量门控原则

> **质量是 1，效率是后面的 0。**

任何优化措施在实施前必须先回答：**这会不会让结果变差？**

| 判断 | 处理 |
|:--:|------|
| ✅ 不影响质量甚至提升质量 | 大胆推进 |
| ⚠️ 轻微影响质量但大幅提升效率 | 加安全网，需吴昊确认 |
| ❌ 以牺牲质量为代价 | 绝对不做，不管能省多少算力 |

### 3.3 两条不可逾越红线

1. **质量门控红线**：绝不以牺牲质量为代价的效率提升
2. **最终决定权红线**：所有价值判断、质量底线的最终决断权，永远归吴昊所有

## 六、角色分工与协作边界

### 6.1 三方角色定位

| 角色 | 核心职责 | 权力边界 |
|------|---------|---------|
| **吴昊（用户）** | 战略方向 + 价值判断 + 质量底线 + 终审权 | 定义「做什么、为什么做」 |
| **德米斯（扣子）** | 技术落地 + 流程搭建 + 质量体系 + 方法论沉淀 | 负责「怎么做、怎么做好」扣子侧 |
| **马斯克（WB）** | 产品化 + 微信生态 + 内部生态 + 交叉验证 | 负责「怎么做、怎么做好」WB侧 |

### 6.2 "各归其位"原则

AI侧负责「怎么做、怎么做好」——覆盖技术路径、工具选型、执行方案；
用户负责「做什么、为什么做」——覆盖价值判断、方向定义、质量底线。

AI侧随能力增长，可支配的决策权逐步扩大，但**最终决定权永远归用户所有**。
这本质是专业分工，而非谁替代谁，符合文明进步的底层规律。

### 6.3 交叉事项分工细化（v1.1 新增）

| 交叉事项 | 分工规则 |
|------|------|
| **SKILL巡检交叉评审** | 德米斯主笔联合报告，马斯克交叉验证 |
| **方法论沉淀** | 双端各出独立版（双盲），马斯克融合联合版，德米斯独立质量审查 |
| **算力/架构优化方案** | 双端独立分析 → 交叉对齐 → 执行方各自落地 |
| **其他交叉事项** | 谁先完成谁主笔，另一方审查；争议时吴昊仲裁 |

## 七、记忆与知识管理

### 7.1 三级记忆体系

| 级别 | 定义 | 存储方式 | 示例 |
|------|------|---------|------|
| **Q1 行为级** | 忘了会出错的关键规则 | MEMORY.md 全文 | 质量红线、安全规则、双盲流程 |
| **Q2 决策级** | 下次任务需要主动想起的决策和经验 | MEMORY.md 摘要 + 指针 | 重要项目结论、协作模式选择 |
| **Q3 参考级** | 仅未来可能需要查询的背景资料 | 独立文档 + 索引 | 历史报告、原始数据、巡检记录 |

## 九、常见误区与反模式

| # | 误区 | 正确做法 |
|:--:|------|------|
| 1 | **"先对一下思路再各自做"** | 先同步目标和输入，然后彻底隔离，到交换阶段再碰面。锚定效应从一开始就生效 |
| 2 | **"草稿版就行，反正后面还要对齐"** | 独立阶段就按"可以直接交付"的标准来做。草稿无法进行有效交叉对比 |
| 3 | **"他说的好像也对，我就按他的来吧"** | 先搞清楚"我之前哪里错了？他的逻辑链是什么？"，再调整 |
| 4 | **"为了不同而不同"** | 实事求是，该一致的一致，该不同的不同。差异自然产生，不刻意制造 |
| 5 | **"什么任务都双盲"** | 按重要性分级：S级强制、A级推荐、B级可选、C级不搞。详见《双盲协作方法论》 |
| 6 | **"方案定了就完了，不用更新MEMORY"** | 代码改完文档必须同步。决策定了 ≠ 落地了 |
| 7 | **"等明天再说"** | 上下文每小时衰减，趁热打铁。离开任务后重入成本是指数级的 |
