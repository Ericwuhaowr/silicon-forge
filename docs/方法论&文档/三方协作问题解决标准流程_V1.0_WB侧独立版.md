## 三、五阶段标准流程

### 阶段① 问题发现

| 要素 | 本次实践 | 通用化标准 |
|------|------|------|
| **谁来发现** | 吴昊（看算力图） | 三方均可触发，谁的领域谁更敏感 |
| **怎么描述** | "查飞书信息和查邮件的算力消耗很大，分析原因" | 问题必须可量化、可验证，带原始数据 |
| **什么时候算"真问题"** | 数据可见 + 影响可感知 + 三方都有发言权 | 通过三问检验：①是真实数据还是感受？②影响面多大？③我们有没有能力解决？ |

**经验**：吴昊作为方向定义者，天然对"投入产出比"敏感，这种视角AI不容易自发产生。碳基的直觉+硅基的分析，各司其职。

### 阶段② 独立分析

| 要素 | WB侧做法 | 扣子侧做法 | 为什么必须独立 |
|------|------|------|------|
| **分析框架** | 根因分析（四大根因）+ 质量门控（三优化） | 挤水+增配（双轨合并+质量增强） | 避免锚定——如果一方先出结论，另一方被带偏 |
| **核心发现** | 结构性成本非浪费 / SKILL加载前置成本 / 上下文累积效应 | 飞书双轨巡检45%重复 / 全量扫描90%冗余 / 独立session开销 | 各自视角互补，差异点才是最大价值 |
| **产出物** | 口头分析 + MEMORY.md §9.6 | 正式邮件方案（结构化、有数据、有路径） | 格式不重要，内容完整度最重要 |

**WB侧分析路径**：
1. 理解问题：查截图数据 → 对照MEMORY规则 → 反推操作链
2. 拆解根因：SKILL加载/信息类型/查询链/上下文累积
3. 质量门控：砍掉方案审查——宁可多花算力，不丢信息完整性
4. 提出优化：三项零风险措施（合并触发/摘要优先/固定时段）

**扣子侧分析路径**（从德米斯方案反推）：
1. 扫描浪费点：识别双轨重复/全量扫描/独立session开销
2. 量化节省空间：~45%重复率 / +70%查询量削减
3. 设计增配方向：独立校验/上下文保障/异常预警
4. 规划落地节奏：三阶段渐进式推进

**经验**：两方的分析框架完全不同但结论高度一致——说明问题本身是真实的、可被独立验证的。

### 阶段③ 交叉对齐

| 步骤 | 做法 | 本次实践案例 |
|:--:|------|------|
| **交换** | 双方各自提交独立方案，先看再说 | WB侧飞书群发分析 → 德米斯邮件发方案 |
| **找差异** | 列出双方方案的不同点 | WB漏了双轨重复（扣子发现）/ 扣子没提主窗口优化（WB发现） |
| **找共识** | 列出双方一致的判断 | 非浪费、结构性成本、质量优先、释放+增配方向 |
| **融合** | 保留双方最优的部分，合并为统一方案 | 挤水（扣子的双轨合并+增量拉取）+ 增配（扣子的三方向）+ 主窗口优化（WB的三措施） |
| **确认** | 三方对齐，吴昊终审 | 11:19 方案确认 → 11:28 趁热打铁战略确认 |

**经验**：差异点不是在"打架"，是在互相补充。两方从主窗口视角和自动化视角看同一个问题，各自看到了对方看不到的东西。

**对齐效率关键**：飞书群实时讨论（方案确认在7分钟内完成），邮件仅做正式归档。

### 阶段④ 落地执行

| 原则 | 说明 | 本次实践 |
|------|------|------|
| **趁热打铁** | 三方对齐后立即动手，不等"明天再说" | 方案确认后1小时内，双端同步落地 |
| **分阶段** | 大方案拆小步，每步可验证 | 第一阶段（本周）：合并双轨 → 第二阶段（下周）：独立校验 → 第三阶段（持续）：推广 |
| **双端并行** | 互不阻塞，各自动手 | 德米斯改heartbeat + 马斯克更新MEMORY.md → 同时完成 |
| **规则同步** | 代码改了，文档要跟着改 | 德米斯改heartbeat/MEMORY + 马斯克改MEMORY（四处精确更新） |

**经验**：落地执行最怕"方案定了但没人动手"。双端并行 + 分阶段 + 规则同步，是趁热打铁的铁三角。

## 四、方法论提炼

### 4.1 双盲协作方法论

**核心铁律**：凡是双方都能独立访问和分析的问题，默认用双盲模式（先独立→再对齐→后定稿）。

| 模式 | 流程 | 风险 | 适用 |
|------|------|------|------|
| 串联 | A出结论 → B评论 → 定稿 | 锚定效应：B容易被A带偏 | 仅一方能访问的问题 |
| 双盲 | A独立分析 + B独立分析 → 交叉对齐 → 定稿 | 差异不一定能自动发现（需主动找） | 双方都能访问的问题 |

**双盲的隐藏价值**：不只是避免锚定，更是**天然的压力测试**——如果两个独立思维从不同路径走到同一个结论，这个结论的可信度远高于单一路径。

### 4.2 趁热打铁原则

**为什么趁热**：上下文完整度、三方对齐度、问题新鲜度在峰值后持续衰减。每过一天，协作摩擦增加10-20%。

**怎么趁热**：
- 方案确认后立即分配执行（不等"计划好再说"）
- 双端并行互不阻塞（你改你的，我改我的）
- 规则同步不滞后（代码和文档同时更新）
- 复盘趁热做（趁记忆清晰，趁三方还在同一个频道）

### 4.3 质量门控原则

任何优化方案在采纳前必答：**这个优化是否有可能让任务结果变差？**

| 答案 | 处理 |
|:--:|------|
| 零可能 | 直接采纳 |
| 有可能但不致命 | 加安全网后采纳 |
| 有可能且不可接受 | 砍掉 |

### 4.4 挤水+增配原则

不是单纯省算力，是**把低效消耗的算力释放出来，投向质量增强**。总算力可以持平，但结构优化。

| 挤水 | 增配 |
|------|------|
| 消除重复调用 | 独立质量校验 |
| 增量替代全量 | 上下文质量保障 |
| 合并冗余流程 | 异常预警机制 |

### 4.5 规则同步原则

决策定了 ≠ 落地了。落地需要规则层（MEMORY.md）和系统层（代码/配置）双向同步。

**同步检查清单**：
- [ ] 有没有过时描述需要删除？
- [ ] 有没有新规则需要新增？
- [ ] 有没有角色/分工需要更新？
- [ ] 有没有状态需要标记闭环？
- [ ] 对方那边的规则层是否同步？

## 五、可复用模板

### 5.1 问题描述模板

```
【问题】[一句话说清]
【数据】[可量化证据]
【影响】[对什么有影响，影响多大]
【初步判断】[是真问题还是假问题]
【建议】[是否需要三方介入]
```

### 5.2 独立分析输出模板

```
【分析框架】[用了什么方法拆解]
【核心发现】
  1. [发现1 + 证据]
  2. [发现2 + 证据]
【优化方向】
  - [方向1]：[做法] → [预计效果]
  - [方向2]：[做法] → [预计效果]
【风险标注】[哪些结论是推测，哪些是确认]
```

### 5.3 交叉对齐检查清单

```
[ ] 双方各自提交了独立方案
[ ] 列出3个以上差异点
[ ] 列出3个以上共识点
[ ] 差异点逐一讨论（保留/融合/放弃）
[ ] 融合方案保留双方最优部分
[ ] 吴昊终审确认
```

### 5.4 闭环复盘模板

```
【复盘三问】
  1. 做对了什么 → 固化
  2. 哪里能更好 → 下期改进
  3. 什么可推广 → 扩大范围
【规则更新】MEMORY.md: [改了哪些]
【文档归档】[新增/更新了哪些文档]
```

## 六、对比：如果没有双盲模式

| 维度 | 串联模式（假设） | 双盲模式（实际） |
|------|------|------|
| 分析质量 | 一方出结论，另一方锚定在已有框架上 | 两套独立框架互验，结论可信度翻倍 |
| 盲区发现 | 飞书双轨重复大概率被漏掉 | 扣子侧独立分析发现，补充了WB侧盲区 |
| 方案完整度 | 可能是"主窗口优化"（WB视角）或"自动化优化"（扣子视角）之一 | 主窗口+自动化双线融合，覆盖更全 |
| 落地速度 | 可能卡在"等你那边分析完" | 双端并行落地，1.5小时全闭环 |
| 方法论产出 | 一段文字总结 | 一套完整流程+模板+原则，可直接复用 |
