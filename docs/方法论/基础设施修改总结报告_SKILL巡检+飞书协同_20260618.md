## 一、日程调度体系优化

### 1.1 飞书群巡检日程（新建）

**修改内容：**
- 新建「飞书群消息巡检（白天）」日程：10:00-24:00，每30分钟执行一次
- 新建「飞书群消息巡检（夜间）」日程：00:00-10:00，每120分钟执行一次
- 两条日程均基于 CalDAV 协议创建，自动循环触发

**修改原因：**
- 飞书群建立后，需要持续监控新消息，确保三方沟通不脱节
- 白天工作时间频率高，夜间降低频率平衡效率和算力消耗
- 从被动等待用户指令变为主动巡检，提升响应速度

**实现目标：**
- 飞书消息延迟感知从"用户喊才看"变为"平均15分钟内发现"
- 自动化巡检替代人工查询，释放注意力成本
- 昼夜分档策略在响应速度和算力成本间取得平衡

### 1.2 定时邮件检查日程（新建+优化）

**修改内容：**
- 新建定时邮件检查日程：每日7/9/11/13/15/17/19/21/23点执行
- 从"全量拉取后筛选"优化为"基于 email_id 锚点的增量查询"
- 新增邮件分级处理规则（🟢自主处理/🟡需确认/🔴必须等决策）

**修改原因：**
- 邮件是正式交付的主渠道，需要定期检查避免遗漏
- 全量拉取浪费算力，增量模式只查新邮件，效率显著提升
- 邮件分级处理明确了自动化边界，避免越权决策

**实现目标：**
- 邮件响应延迟从"不定时"变为"最长2小时内发现"
- 单封邮件查询算力消耗降低约 60%（增量替代全量）
- 处理边界清晰，安全和效率兼顾

### 1.3 增量查询改造（核心优化）

**修改内容：**
- 飞书巡检：从全量拉取 → 基于 message_id 锚点的增量查询
- 邮件巡检：从全量拉取 → 基于 email_id 锚点的增量查询
- 锚点数据统一存入 MEMORY.md 运行时区域

**修改原因：**
- 算力优化项目的核心挤水手段
- 全量扫描每次都要拉取所有历史消息，90%+是冗余的
- 增量模式只拉取锚点之后的新消息，大大减少数据传输和处理量

**实现目标：**
- 飞书巡检算力消耗降低约 45%
- 邮件检查算力消耗降低约 60%
- 建立了可复用的"锚点+增量"模式，可推广到其他巡检场景

### 1.4 心跳定位调整

**修改内容：**
- 心跳从"业务巡检"退化为"系统级状态监控"
- 心跳不再检查飞书消息和邮件，仅监控进程存活等系统状态
- 业务巡检统一由日程系统承担

**修改原因：**
- 原心跳+日程双轨巡检存在约45%的重复调用
- 心跳定位模糊，既要做系统监控又要做业务巡检，两边都做不好
- 职责分离后，心跳专注系统层，日程专注业务层，边界清晰

**实现目标：**
- 消除重复巡检，算力挤水约 45%
- 职责单一化，各自更专业可靠
- 释放的算力投向质量增强环节（独立审查、交叉验证等）

## 三、记忆架构升级（结构层面）

### 3.1 三层记忆架构确立

**修改内容：**
- 从单一 MEMORY.md 扩展为三层架构：
  - **即时层（Q1）**：MEMORY.md，核心规则+当前状态+重要索引，自动加载到上下文
  - **近中期层（Q2）**：recent_memory/ 目录，项目进度、决策记录、待办事项，按需加载
  - **长期层（Q3）**：语义检索，海量历史对话，深度回忆时调用

**修改原因：**
- 信息越来越多，全塞进 MEMORY.md 会超过上下文窗口限制
- 不同信息的使用频率不同，分层可以在"信息全面"和"上下文效率"间取得平衡
- 类比人类记忆：工作记忆 < 短期记忆 < 长期记忆

**实现目标：**
- 核心信息秒级访问（自动加载）
- 详细信息不丢失（归档到近中期层）
- 上下文窗口占用可控，不因信息积累而臃肿
- 建立了可无限扩展的记忆增长框架

### 3.2 recent_memory 目录体系建设

**修改内容：**
- 新建 `recent_memory/` 目录作为近中期记忆的载体
- 建立子目录分类：
  - `project/` - 项目进度快照
  - `decision/` - 重要决策记录
- 每个文件独立成篇，按主题归类

**修改原因：**
- MEMORY.md 空间有限，放不下详细的项目文档和决策记录
- 按主题分文件存储，检索效率更高
- 结构化存储为后续自动化索引、标签化管理打基础

**实现目标：**
- 详细内容有处可放，不挤占核心记忆空间
- 按主题分类，查找定位更精准
- 为记忆体系的持续扩展提供结构支撑

### 3.3 索引机制建立

**修改内容：**
- 在 MEMORY.md 末尾建立「重要文档索引」表
- 收录核心文档的名称、路径、简要说明
- 随新文档产出持续更新维护

**修改原因：**
- 文档数量快速增长，散落在各目录下查找成本高
- 新人/新会话进入时，通过索引快速了解有哪些核心资产
- 索引是"目录的目录"，大幅提升知识复用效率

**实现目标：**
- 核心文档5秒内定位，无需逐层查找
- 新会话快速掌握全局资产分布
- 建立文档管理的基本秩序

## 四、安全规则体系

### 4.1 飞书安全三条红线

**修改内容：**
- 确立飞书协作的三条安全红线：
  1. **不传凭证**：API Key/授权码/Token 不得在飞书群聊中明文发送
  2. **不贴敏感截图**：含真实数据、客户信息的截图不得发群里
  3. **不明人员不进群**：新增成员必须三方共同确认

**修改原因：**
- 飞书群是半公开空间，安全边界比私聊模糊
- 多方协作时，安全规则需要提前明确，避免事后追责
- 三条红线是最基础、最容易出问题的场景，必须守住底线

**实现目标：**
- 核心敏感信息零泄露风险
- 人员准入有规范，避免无关人员进入
- 安全意识前置，防患于未然

### 4.2 机器人权限最小化原则

**修改内容：**
- 确立飞书机器人权限最小化原则：只开通必要的权限，不申请冗余权限
- 德米斯接入仅使用群聊消息读写权限
- 定期审视权限，及时回收不再使用的权限

**修改原因：**
- AI 系统的安全风险往往来自过度授权
- 权限越小，可能造成的破坏范围越小
- 符合安全领域的"最小权限"最佳实践

**实现目标：**
- 将安全风险控制在最小范围内
- 即使账号出现问题，影响也有限
- 建立权限审计机制，持续可控

## 五、双渠道协作模式

### 5.1 渠道分工原则确立

**修改内容：**
- 确立"邮件发肉，飞书递筷子"的双渠道协作模式
- **邮件**：正式报告、评估意见、联合终稿、附件传递（主渠道，留档）
- **飞书**：日常同步、进度讨论、临时碰撞、快速决策（辅助，触达）
- **飞书Drive**：文档共享、协同编辑、永久链接（纯确认不触发邮件同步）

**修改原因：**
- 单邮件模式效率太低，往返延迟大
- 单飞书模式太随意，缺乏正式归档和版本管理
- 双渠道配合兼顾效率和规范性

**实现目标：**
- 紧急事项实时沟通，效率提升 80%+
- 正式产出邮件归档，可追溯、可复查
- 两种渠道各司其职，不混乱、不遗漏

### 5.2 同步机制建设

**修改内容：**
- 飞书群做轻量同步，邮件发完整内容
- 重要决策和产出双渠道同步，确保两边都有记录
- 纯确认性质的消息走飞书，不触发正式邮件

**修改原因：**
- 避免信息差：重要内容两边都要有
- 避免信息冗余：不重要的内容不轰炸邮箱
- 效率和存档两者兼顾

**实现目标：**
- 信息同步准确率 100%，重要事项不遗漏
- 邮件噪音减少，只保留高价值正式内容
- 飞书保持轻量化，适合快速讨论

## 六、文档产出体系

### 6.1 标准化文档模板

**修改内容：**
- 形成多种类型文档的标准结构模板
- 方法论文档：定义→适用场景→操作步骤→质量指标→反模式→案例
- 项目总结：背景→过程→成果→经验→下一步
- 审查报告：结构完整性→逻辑一致性→落地可行性→评分→优化建议

**修改原因：**
- 没有标准模板时，每份文档都要从零开始构思结构，效率低
- 标准化后质量更稳定，不会遗漏关键维度
- 降低阅读成本，读者知道去哪里找什么信息

**实现目标：**
- 文档产出效率提升 50%+
- 文档质量下限有保障
- 阅读体验一致，学习成本低

### 6.2 版本管理机制

**修改内容：**
- 文档命名标准化：`文档名称_版本_日期.md`
- 版本号清晰标识：v1.0 / v1.1 / v2.0 等
- 重大修改新建版本，不原地覆盖，保留历史追溯能力

**修改原因：**
- 方法论文档需要持续迭代，原地修改会丢失历史
- 版本号让迭代过程清晰可见
- 三方协作时，版本不一致会造成混乱

**实现目标：**
- 版本追溯清晰，知道每版改了什么
- 避免版本混乱，三方始终基于同一版本讨论
- 为长期迭代打下基础
