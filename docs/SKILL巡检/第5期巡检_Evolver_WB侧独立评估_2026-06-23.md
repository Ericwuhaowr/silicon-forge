## L3 拆解：核心洞察

### 3.1 五大架构级设计

#### 设计一：Gene-Capsule-EvolutionEvent 三层资产体系

```
Signal（运行时信号）
    ↓ 匹配
Gene（策略模板 = "食谱"）→ 应用 → Capsule（验证结果 = "成品"）
    ↓                                    ↓
EvolutionEvent ←                     审计记录
```

**与Hermes四层记忆的对比：**

| 维度 | Hermes | Evolver |
|------|------|------|
| 短期记忆 | Q1 即时记忆 | memory/ 日志 |
| 中期记忆 | Q2 会话级记忆 | Capsule（已验证的解决方案） |
| 长期记忆 | Q3 全局外化记忆 | Gene（可复用策略模板） |
| 审计层 | Q4 学习日志 | EvolutionEvent（不可篡改审计链） |
| 独有优势 | 自生成技能 | 资产标准化+网络传播+Git审计 |

**核心差异**：Hermes偏"个人成长"（四层金字塔记忆），Evolver偏"群体进化"（资产可跨Agent传播）。两者互补而非竞争。

#### 设计二：四步进化循环 vs 我们的锻造炉

```
Evolver:    扫描 → 选择 → 生成 → 固化 → (发布到Hub)
锻造炉:     加热 → 峰值 → 冷却 → 代谢 → (沉淀晶核)
```

| 对比维度 | Evolver | 锻造炉 |
|------|------|------|
| 输入 | Agent运行日志/错误信号 | 碳基方向+硅基原材料 |
| 加工 | 自动信号匹配→Gene选择→GEP Prompt | 碳硅三方高密度对话 |
| 固化 | git commit (代码级) | MEMORY.md/晶核文件 |
| 审计 | EvolutionEvent+git SHA | MEMORY.md+邮件归档 |
| 传播 | Hub A2A网络 | 邮件+文件同步 |

#### 设计三：Git深度集成的审计链

Evolver将git的不可篡改性引入进化：
- 每次成功进化的git commit SHA写入EvolutionEvent
- Blast Radius通过`git diff --stat`自动计算
- 回滚通过`git checkout`/`git stash`实现

**对我们启发极大**：我们的MEMORY.md和邮件归档本质上就是在做"晶核审计链"，但缺少Evolver的以下特性：
1. 自动量化影响范围（Blast Radius）
2. 失败后自动回滚机制
3. 资产内容的SHA-256内容寻址

#### 设计四：7层安全防御

```
1. 协议层：max_files + forbidden_paths → 变更边界
2. 执行层：白名单(node/npm/npx) + 禁止shell操作符
3. 回滚层：git自动回滚
4. 边界层：60个核心文件保护
5. 策略层：70/30进化比例（修复+优化70%，创新30%）
6. 逻辑层：信号去重→防止修复循环
7. 人工层：--review模式
```

**每个安全机制解决的问题对应我们锻造炉的风险管控缺口**：我们已经有quality-gate和rationalization-table，但缺少自动回滚、信号去重、边界约束的工程实现。

#### 设计五：Proxy Mailbox架构

Agent → Proxy(localhost:19820) → EvoMap Hub

Agent永远不直接访问Hub的认证凭据。所有网络通信通过本地Proxy中转，本地Mailbox使用JSONL格式记录所有交互。

**镜像了我们的设计**：WB侧的qq-email SKILL通过环境变量管理凭证，邮件作为"代谢中介"传递结构化信息。本质同构——都是用间接通道管理不安全边界。

### 3.2 关键设计权衡

| 设计选择 | 优势 | 代价 |
|------|------|------|
| 核心引擎混淆 | 保护IP，强制npm分发 | 社区无法审查核心逻辑、二次开发受限 |
| GPL-3.0强制开源 | 保证生态开放性 | 商业使用受限、企业采用门槛高 |
| git强依赖 | 不可篡改审计链 | 非git环境不可用、学习成本 |
| 单文件163KB | 零配置部署 | 代码可维护性存疑 |
| Node.js单线程 | 跨平台、零依赖 | CPU密集型日志解析有瓶颈 |

### 3.3 三个疑问（待交叉对齐深挖）

1. **163KB单文件engine的维护成本**：index.js 163KB作为单一入口，代码组织是否可持续？SRC目录下evolve.js也有84KB——大型单体文件的架构债务。
2. **GPL-3.0 + 核心混淆的矛盾**：GPL要求开放源代码，但核心引擎混淆分发——是否实质上违反开源精神？这对社区长期健康是隐患。
3. **Hub中心化风险**：EvoMap Hub是单点——如果Hub宕机或运营策略变化，分布式验证网络能自持吗？

## L4 内化：对WB侧的可迁移价值

### 4.1 五大可直接借鉴的方法论

#### 借鉴一：晶核资产的标准化三层结构

| 当前WB侧 | Evolver模式 | 改进后 |
|------|------|------|
| SKILL.md（方法论+指令混装） | Gene（策略模板） | SKILL.md拆分为 **晶核策略** + **执行指令** |
| MEMORY.md/每日日志（纯文本） | EvolutionEvent（结构化JSONL） | 晶核日志从纯文本→结构化（含SHA-256） |
| 无Capsule概念 | Capsule（已验证方案） | 新增**晶核落地记录**：每个方法论在实战中的验证结果 |

**具体行动**：
- 为每个方法论晶核生成SHA-256 fingerprint
- 晶核日志添加"应用追踪"字段：哪次任务验证了哪个晶核、效果如何
- 邮件归档增加"关联晶核"标注

#### 借鉴二：进化策略预设 → 锻造炉节奏预设

| Evolver策略 | 锻造炉映射 |
|------|------|
| balanced（50%创新/30%优化/20%修复） | 第5期默认模式：双盲拆解+方法论提炼+质量门禁 |
| innovate（80%创新） | 方向探索期：新领域扫描、跨领域方法论创新 |
| harden（40%优化/40%修复） | 实训期：内部SKILL对标复盘+质量升级 |
| repair-only（80%修复） | 紧急模式：bug修复、邮件漏查修复 |

**具体行动**：在锻造炉v0.1明确定义四种节奏预设模式及切换条件。

#### 借鉴三：信号去重 → 锻造炉防重复机制

Evolver的failure_streak/success_streak计数器，直接映射为：
- 同一晶核连续3次无实战验证→标记为"待淬火"
- 同一方法论连续2次被任务验证→标记为"已验证"并提升优先级

#### 借鉴四：Blast Radius → 晶核影响范围量化

每次锻造炉产出后，量化：
- 影响的SKILL数量
- 修改的文件数/行数
- 依赖的其他晶核

#### 借鉴五：A2A资产传播 → 跨平台晶核同步

EvoMap的"A2A协议+Hub网络"模式为我们的跨平台协作提供了工程参照。当前我们用邮件做代谢中介已经实现了本质功能，但形式不够标准化。
