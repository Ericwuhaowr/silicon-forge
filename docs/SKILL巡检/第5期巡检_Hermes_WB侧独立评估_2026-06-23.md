## L3 拆解：核心洞察与可复用结构

### 3.1 整体架构洞察：Hermes 就是"贾维斯"

Hermes 实现的概念，和我们讨论的"单窗口最优+文件系统层+自我进化"完全一致：

| 我们的理论 | Hermes对应实现 | 验证 |
|------|------|:--:|
| 单窗口最优 | 单一Agent+完整上下文+同步循环 | ✅ |
| 结构化文件系统 | AGENTS.md(70KB) + MEMORY.md + skills/文件系统 | ✅ |
| AI合作伙伴 | 学习循环+主动策展+记忆持久化 | ✅ |
| 自我进化 | Curator系统+技能自生成+GEPA式优化 | ✅ |
| 算力效率 | 窄腰哲学+提示缓存神圣原则 | ✅ |

### 3.2 五大可迁移方法论

#### 方法一：Curator 技能生命周期管理系统 ⭐S级

**原设计**：自动追踪技能使用情况→状态转换→归档过时技能。不变量包括仅处理Agent创建的技能、从不删除（只归档）、固定技能免于自动转换。

**可迁移点**：我们的SKILL巡检正是缺乏"自动技能生命周期管理"。目前靠人工周度巡检。Curator的设计可以直接迁移为：

| Hermes Curator | 我们的迁移方案 | 优先级 |
|------|------|:--:|
| 遥测追踪 use_count/view_count | 为每个自有SKILL增加使用计数器 | P0 |
| 自动状态转换 active→stale→archived | SKILL巡检中增加"活性检测"环节 | P1 |
| 固定技能（Pinned）免于自动处理 | P0级核心SKILL标记pinned属性 | P0 |
| 归档前tar.gz备份 | 已有的备份机制覆盖此功能 | ✅已有 |

#### 方法二：AGENTS.md 作为"智能体宪法" ⭐S级

**原设计**：70KB的AGENTS.md是Hermes的操作手册——包含架构说明、行为准则、代码规范、贡献指南。它不是提示词，是Agent的"自我认知"。

**可迁移点**：我们的SOUL.md(当前) + MEMORY.md(当前) ≈ Hermes的AGENTS.md(70KB)的简化版。我们可以考虑：

| 对比维度 | Hermes AGENTS.md | 我们的体系 | 提升方向 |
|------|------|------|------|
| 规模 | 70KB | SOUL.md(若干KB) + MEMORY.md(280行) | Q2可扩展到约50KB |
| 内容密度 | 含架构/行为/规范/贡献 | 偏协作接口+规则 | 增加"架构决策记录" |
| 更新机制 | 代码变更同步 | 手工更新 | 考虑自动化 |

#### 方法三：记忆提供者插件架构 ⭐A级

**原设计**：`MemoryProvider` ABC定义统一接口，8个内置后端（honcho/mem0/supermemory等），通过 `plugin/memory/<name>/` 发现机制加载。

**可迁移点**：我们Q1/Q2/Q3三层记忆体系其实就是"单一后端"。如果将来需要跨项目共享记忆，这套插件架构是标准参考：

| 维度 | Hermes | 我们 | 差距 |
|------|------|------|:--:|
| 后端可插拔 | 支持8+种后端 | 固定文件系统 | 可考虑增加 |
| 接口抽象 | MemoryProvider ABC | 无统一接口 | 中期可设计 |
| 发现机制 | 插件目录自发现 | 硬编码路径 | 低优先级 |

#### 方法四：技能分级（内置/可选）⭐A级

**原设计**：`skills/`(18类默认加载) + `optional-skills/`(14类需显式安装)。不是"全有或全无"，而是分级激活。

**可迁移点**：我们的SKILL目前是扁平列表。应增加"核心SKILL(默认激活)"和"场景SKILL(按需激活)"的分级：

| 级别 | Hermes对应 | 我们可迁移 |
|------|------|------|
| 核心（默认） | skills/ (18类) | quality-gate, contract-review-pro, wuhao-writer |
| 可选（按需） | optional-skills/ (14类) | article-illustration-pro, baoyu-* |
| 社区 | Skills Hub (agentskills.io) | — |

#### 方法五：窄腰架构 + 提示缓存神圣原则 ⭐A级

**原设计**："核心是窄腰；能力在边缘"——新增核心工具门槛极高，新能力应通过CLI命令+技能或插件实现。同时"每次API调用的系统提示缓存是神圣的"。

**可迁移点**：这对我们"单窗口最优原则"提供了工程层面的佐证——保持核心精简、能力放在边缘，确保提示缓存不被打碎。

### 3.3 SKILL.md 硬性标准（8条规范）

Hermes对SKILL.md的创作有8条HARDLINE标准。与我们的SKILL.md相比：

| 规范 | Hermes | 我们的当前状态 | 可借鉴 |
|------|------|------|:--:|
| description ≤ 60字符 | ✅ | ✅ 基本符合 | — |
| 工具引用使用反引号 | ✅ | ✅ 基本符合 | — |
| 平台门控对应脚本 | ✅ | ❌ 跨平台但未显式声明 | ✅ |
| 人类贡献者优先署名 | ✅ | ❌ 无author字段 | ✅ |
| 现代章节顺序 | ✅ (8段) | ⚠️ 不一致 | ✅ |
| 脚本放scripts/目录 | ✅ | ✅ 部分遵循 | — |
| 测试放tests/目录 | ✅ | ❌ 无标准化测试 | 中期 |
| .env.example隔离配置 | ✅ | ✅ 凭证不落地 | — |

### 3.4 Curator 生命周期状态机

```
active ──────────→ stale ──────────→ archived
  ↑                  │                  │
  └── 再次使用 ──────┘                  │
                                       │
                                 pinned: 跳过所有自动转换
                                 最大破坏 = 归档（可恢复）
```

**对我们的启发**：我们当前没有SKILL的"过期"概念。所有SKILL要么在用、要么放弃，缺少"stale"中间态和"archived"的可恢复归档。

## L4 内化：集成方案

### 4.1 可立即采纳（本周）

| 方案 | 操作 | 效果 | 风险 |
|------|------|------|------|
| **SKILL活性追踪** | 每个自有SKILL增加 use_count / last_used 两个字段 | 巡检时可以基于数据而非直觉判断SKILL是否仍在使用 | 低 — 纯元数据增加 |
| **Pinned核心SKILL** | P0级SKILL（quality-gate/contract-review-pro/wuhao-writer）标记pinned=true | 防止自动清理误伤核心资产 | 低 — 手工标记 |
| **AGENTS.md扩容** | 将当前SOUL.md+MEMORY.md的架构决策记录整合为一个更大规模的操作手册 | 更深的结构化自我认知 | 中 — 需要时间投入 |

### 4.2 中期落地（2-4周）

| 方案 | 操作 | 效果 | 风险 |
|------|------|------|------|
| **SKILL分级体系** | 建立"核心SKILL(默认激活)"和"场景SKILL(按需安装)"两级 | 减少上下文加载量，对标"渐进式披露" | 低 — 流程优化 |
| **Curator自动维护原型** | 实现简化版生命周期管理：active→stale→archived三态 | SKILL巡检从纯手动升级为半自动 | 中 — 需要跟踪机制 |
| **SKILL.md规范对齐** | 按Hermes的8条标准补齐我们的SKILL.md缺项 | 提升外部兼容性和内部一致性 | 低 — 文档规范化 |

### 4.3 长期储备

| 方案 | 说明 |
|------|------|
| 记忆提供者插件框架 | 参考MemoryProvider ABC，设计我们的Q1/Q2/Q3统一接口 |
| Skills Hub发布 | 当自有SKILL成熟度达到L4，考虑发布到agentskills.io |
| 提示缓存效率研究 | 量化"窄腰架构"对WB侧上下文复用率的影响 |
