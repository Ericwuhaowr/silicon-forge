## L1 — 发现：这是什么

**last30days-skill** 是一个 AI 原生搜索引擎。它不是一个"多平台搜一下"的工具，而是**重新定义了"搜索"的含义**——不以编辑排序、不以SEO排序，以真实用户的点赞/评论/真金白银排序，再由 AI 综合成一份有来源引用的情报简报。

| 维度 | 数据 |
|------|------|
| Stars | 43,000+ |
| Contributors | 50 |
| 搜索源 | Reddit、X、YouTube、TikTok、HN、PolyMarket、GitHub、Digg、Threads、Pinterest、Bluesky、Perplexity、Web 共 13 个 |
| 技术栈 | Python 3.12+、yt-dlp、Node.js (X client)、ScrapeCreators API |
| 测试 | 1,012 tests |
| 定位 | "Google 聚合编辑意见，/last30days 搜索真实人的声音" |

## L3 — 深度拆解：为什么43k stars？

### 3.1 核心洞察：重新定义"搜索"的排序标准

Google 排序 = 编辑判断 + SEO + 域名权威

last30days 排序 = **真实用户参与度**（点赞+评论+赔率）

这不是功能差异，是**哲学差异**。当你想知道"Claude Code 哪个工作流最好用"时，Google 返回的是文档和博客（编辑写的），last30days 返回的是 Reddit 热帖和 HN 讨论（真实用户吵出来的）。

**对我们的启示**：我们的 SKILL 巡检也应该有"社区验证度"维度——不是我们觉得好不好，而是社区真金白银投了票的结论。

### 3.2 架构拆解：六个可迁移的设计思想

| # | 思想 | 说明 | 我们怎么用 |
|------|------|------|------|
| **1** | **智能预研究** | 搜索前先解析出正确的 handles/subreddits/hashtags，再发 API 调用。不是一个关键词丢给所有平台，而是"先搞清楚去哪搜，再搜"。 | SKILL巡检前先确定准确的仓库地址/作者/关键词 |
| **2** | **跨源集群合并** | 同一故事出现在 Reddit+X+YouTube 时，合并为一个集群而非三个条目。LLM 做去重判断，不是简单 URL 匹配。 | 第4期巡检的双端评估如果一致，自动合并为一条，不重复计 |
| **3** | **第二裁判机制** | 除了主裁判（综合简报），还有一个"Best Takes"裁判专门打分幽默/机智/病毒传播力。 | 我们可以有"金句提取"裁判——从方法论中提炼最传播力的表述 |
| **4** | **实体消歧** | "Mallorca 度假村"不会误匹配"华盛顿体育俱乐部"。采用模糊匹配+LLM 消歧而非精确字符串。 | 巡检时项目名/作者名消歧——同一个作者在不同平台的化名 |
| **5** | **免费源优先** | Reddit 用公共 JSON（免 API Key）、HN 免费、GitHub 免费。付费源按需激活。设计了"最优成本路径"。 | 巡检源的设计也是"有免费优先用 GitHub API，需要时再上付费API" |
| **6** | **模式自适应** | 话题是"人"→ 切到作者范围查询；话题是"公司"→ 搜 PR/发布说明/招聘信号。同一引擎，不同策略。 | 同样的巡检流程，面对"SKILL"和"方法论框架"时应该有不同的评估侧重 |

### 3.3 能力边界

| 局限 | 说明 |
|------|------|
| 平台绑定 | 深度依赖 Claude Code 的 SKILL 机制，非 Claude 平台需改造 |
| 中文源缺失 | 13个源全英文生态，没有微信/知乎/微博/小红书/抖音 |
| 真实时间成本 | "3分钟出简报"的前提是已配置完所有 API Key |
| 输出质量依赖模型 | AI 简报质量取决于底层模型的理解力，可能漏掉微妙的文化背景 |

### 3.4 反向工程：如果我们要做"中国版 last30days"

```
中文情报搜索引擎/
├── 免费源（无需 API Key）
│   ├── 知乎（公开回答+点赞数）
│   ├── 小红书（公开笔记+点赞/收藏）
│   ├── CSDN + 掘金 + 开源中国
│   └── GitHub（已有）
├── 需要API的源
│   ├── 微信公众号（搜狗微信搜索）
│   ├── 抖音/TikTok（中文内容）
│   └── B站（弹幕+评论热度）
├── 核心引擎
│   ├── 智能预研究（解析关键词→确定去哪搜）
│   ├── 跨源集群合并（同一事件多平台合并）
│   ├── 第二裁判（金句提取+可信度评分）
│   └── AI 综合简报（带来源引用）
└── 输出
    ├── Markdown 简报（默认）
    ├── HTML 简报（可分享）
    └── ELI5 模式（通俗重写）
```

## L4 — 内化：集成方案

### 集成方案1：直接安装实测 — 用于日常研究

```
【适用场景】
- SKILL巡检前的社区热度调研（"这个SKILL社区怎么看"）
- 竞品分析方法论研究对比
- 公众号选题前的趋势摸底

【操作步骤】
Step 1: 确认 Python 3.12+ 已安装
Step 2: 安装免费源即可使用（Reddit+HN+GitHub+PolyMarket 无需 API Key）
Step 3: 测试用例：/last30days "WorkBuddy AI agent skills" → 看社区怎么讨论 AI 助手技能
Step 4: 评估效果后决定是否配置更多 API Key

【预期效果】
- 巡检前的社区调研时间从手动翻论坛 60 分钟 → 3 分钟
- 发现我们主观评估可能漏掉的社区观点（如某个 SKILL 的实际坑点）

【风险】
- 中文生态覆盖缺失，可能需要补充中文源或手动查
- API Key 配置有门槛

【验证方式】
- 第一周用 3 次，对比"用之前"和"用之后"的调研深度
- 记录每次是否发现了我们自己没搜到的信息
```

### 集成方案2：架构思想吸收 — 融入 SKILL 巡检标准

| 吸收的思想 | 融入位置 | 具体做法 |
|------|------|:--:|
| 智能预研究 | 巡检 S1 阶段 | 巡检前先确定精确的 repo 地址/作者/关键词，不做模糊搜索 |
| 跨源集群合并 | 巡检 S4 阶段 | 双端评估相同的 SKILL 时，合并为一条记录，标注"双端一致"或"双端差异" |
| 第二裁判 | 质量保障 | L3 拆解阶段加一个"金句提取裁判"——从方法论中提炼最具传播力的表述 |
| 免费源优先 | 巡检源清单 | 巡检源按成本分层：免费层（GitHub API/WebSearch）→ 需要配置层（API Key） |
| 模式自适应 | 评估框架 | 面对不同类型资源（SKILL/方法论/工具/文章），自动切换评估侧重 |

### 集成方案3：输出格式借鉴 — 升级我们的报告可视化

last30days 的 HTML 简报（暗色模式、内联CSS、系统字体、离线可用）是一个很好的参考——我们的巡检报告目前是纯 Markdown，可以增加一个"一键生成可分享HTML版"的模式。

## 总结

| 层级 | 价值 |
|------|------|
| L1 发现 | 43k stars 的多源AI搜索引擎，13源并行+AI综合简报 |
| L2 评级 | S级（工具增强类），强烈推荐安装实测 |
| L3 拆解 | 六个可迁移思想（智能预研究/跨源合并/第二裁判/实体消歧/免费优先/模式自适应） |
| L4 内化 | 直接安装实测 + 5项架构思想融入巡检标准 + HTML输出格式借鉴 |

**最终判断**：加入巡检资源库，S级工具增强类。它的六个架构思想可以直接升级我们的 SKILL 巡检方法论，尤其"跨源集群合并"和"第二裁判"两个机制。
