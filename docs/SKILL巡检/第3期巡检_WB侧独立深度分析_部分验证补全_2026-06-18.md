## SKILL 1 · neat-freak V2 — 记忆防膨胀工具

### 独立研究路径

- 详细文档: [skills.cat/neat-freak](https://skills.cat/skills/kkkkhazix/khazix-skills/neat-freak)
- 腾讯云中文介绍: https://cloud.tencent.com/developer/article/2685328

### 核心发现

**定位**：会话结束后的知识清理工具（不是"记录工具"，是"编辑工具"）

**三层受众体系**（这是精妙之处）：

| 层级 | 受众 | 作用 |
|------|------|------|
| Agent记忆（MEMORY.md等） | AI自己跨会话 | 个人偏好、非显见事实 |
| 项目根CLAUDE.md/AGENTS.md | 本项目下一个AI | 项目约定、结构、红线 |
| docs/ + README | 人类同事+下游开发者 | 接入指南、架构、运维手册 |

**五步执行流程**：
1. 机械式枚举（ls所有文件，不漏一个）
2. 变更影响矩阵（新事实→波及哪些文档层）
3. 实际修改（用Edit/Write工具，不是描述性建议）
4. 自检清单（11项逐条打勾）
5. 变更摘要（向用户汇报）

**硬核约束**：
- 合并优于追加、删除优于保留、精确优于冗长
- 绝对时间（2026-04-29）不写"今天"
- 全局配置极度克制
- 跨项目影响必须对齐

### WB侧独立评级：S/P0

**理由**：
1. 和我们wuhao-neat-freak-防膨胀互补——我们偏"文档瘦身"，它偏"知识体系全局同步"
2. 三层受众体系直接解决我们MEMORY.md分级策略中的"写给谁看"问题
3. 变更影响矩阵和我们刚定的Q1-Q3分级逻辑一致
4. 自检清单可以直接作为MEMORY.md更新后验证的标准

**借鉴路径**：
- 不整体引入（卡兹克生态SKILL，WB需适配），但提取三层受众体系和变更矩阵方法论
- 和wuhao-neat-freak合并升级为v2.0

## SKILL 2 · Firecrawl — 网页数据采集引擎

### 独立研究路径
- GitHub: [mendableai/firecrawl](https://github.com/mendableai/firecrawl)
- 官网: https://firecrawl.dev
- 许可: AGPL-3.0（核心）/ MIT（SDK）

### 核心发现

**完整能力矩阵**：

| 能力 | 说明 |
|------|------|
| Search | 搜索网页+获取完整内容 |
| Scrape | URL→Markdown/HTML/截图/JSON |
| Crawl | 单次请求爬取整个网站 |
| Map | 发现网站所有URL |
| Batch | 异步批量抓取数千URL |
| Agent | AI代理自动化采集（无需指定URL） |
| Interact | 点击/滚动/输入/等待后再提取 |

**性能**：P95 3.4秒延迟，覆盖96%网页（含重度JS）

**集成方式**：Python/Node.js/Java/Elixir/Rust/Go SDK + CLI + MCP客户端（开箱即用）

**与WB侧巡检的场景匹配**：

| 巡检步骤 | Firecrawl如何使用 |
|------|------|
| SkillHub排名扫描 | Scrape→提取结构化数据 |
| GitHub新SKILL发现 | Search→获取README |
| 外部文章/论文采集 | Scrape→Markdown→嵌入分析 |
| 多源并行巡检 | Batch Scrape批量处理 |

### WB侧独立评级：S/P0

**理由**：
1. 巡检自动化数据采集层有真实缺口——目前靠手动WebSearch+WebFetch，效率低
2. 7种SDK+MCP开箱即用，接入成本极低
3. Agent模式（描述需求自动采集）对定期巡检非常实用
4. 唯一顾虑：AGPL-3.0许可，但SDK是MIT，作为工具使用不受影响

**借鉴路径**：
- P0：直接作为巡检数据采集工具接入（MCP或Python SDK）
- 和search-specialist对比：Firecrawl偏"执行层"（真的去爬），search-specialist偏"方法层"（怎么搜得更好）——两者互补而非替代

**延伸阅读**：
- Firecrawl官方文档: https://docs.firecrawl.dev
- MCP集成指南: https://docs.firecrawl.dev/integrations/mcp

## SKILL 3 · Context-Engineering — 上下文工程框架

### 独立研究路径
- GitHub: [muratcankoylan/Agent-Skills-for-Context-Engineering](https://github.com/muratcankoylan/Agent-Skills-for-Context-Engineering)
- ScriptByAI Top10分析: https://www.scriptbyai.com/best-agent-skills/

### 核心发现

**"注意力预算"管理框架**：把AI上下文当作有限资源来管理。

**核心组件**：
1. 多代理架构设计（任务分解→独立上下文→结果汇总）
2. 上下文压缩策略（蒸馏、剪枝、分层）
3. 评估框架构建（量化上下文使用效率）
4. 沙盒化后台代理（长任务独立运行→只回传结果）

**与WB侧8项优化方案的对照**：

| WB侧8项方案 | Context-Engineering对应概念 |
|------|------|
| 工具结果剥壳 | 上下文压缩策略 |
| Skill要点复用 | 分层加载 |
| 日程分级回灌 | 注意力预算分配 |
| 骨架先行 | 评估框架 |
| 记忆分层 | 分层存储 |
| 并行调用 | 沙盒化代理 |
| Agent子代理（已否决） | 多代理架构 |

**关键发现**：WB侧8项方案中6项有理论对应，说明我们的优化方向正确。但我们否决的"Agent子代理"在这个框架中是通过"沙盒化后台代理"实现的——不做决策只执行，和我们之前担心的"脱离上下文盲干"不同。

### WB侧独立评级：S/P0

**理由**：
1. 理论框架和WB侧实践高度吻合，确认了我们的优化方向
2. "沙盒化后台代理"提供了一种安全的子代理模式——值得我们重新评估之前否决的Agent子代理决策
3. 评估框架可以直接用于量化我们MEMORY.md分级的效果

**借鉴路径**：
- P0：对照验证8项优化方案的理论覆盖度
- 重新评估Agent子代理在"沙盒化"约束下的可行性

**延伸阅读**：
- ScriptByAI Top 10: https://www.scriptbyai.com/best-agent-skills/
