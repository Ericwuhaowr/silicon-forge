## L1 — 发现：这是什么

**claude-code-best-practice** 不是一个传统的 SKILL——它是一个 **"Agentic Engineering 知识地图"**。作者 shanraisshan 用不到 60 天时间，将一个 GitHub README 文件做成了 AI 工程社区的现象级项目。

| 维度 | 数据 |
|------|------|
| Stars | 54.1k（不到60天） |
| 核心内容 | 83条实操技巧 + 12个完整工作流 + 8个技能集合 + 跨模型桥接方案 + "十亿美元问题" |
| 定位 | from vibe coding to agentic engineering — 从"凭感觉写代码"到"系统化AI工程" |
| 组织方式 | 按原语组织：Subagents → Commands → Skills → Workflows → Hooks → MCP → Memory |

## L3 — 深度拆解：为什么能爆到 54k stars？

### 3.1 核心洞察：解决了 Agentic Engineering 的三个元问题

这个项目的真正价值不在"内容本身"，而在它回答了三个根本问题：

| 元问题 | 它的回答方式 | 我们的同类问题 |
|------|------|------|
| **什么时候用什么工具？** | 用 Command / Agent / Skill 决策树精确区分三者的适用边界 | 什么时候用能力跟随决策权 / 什么时候用双盲对齐？ |
| **怎么把AI工程系统化？** | 提炼出 Research→Plan→Execute→Review→Ship 五阶段通用模式，12个工作流全部遵循 | 我们的五步法（0-4阶段）是否足够通用？ |
| **社区的共识是什么？** | 通过量化社区选择（star数+安装量）告诉你"什么方案被广泛验证过" | 我们如何系统性地追踪外部方法论？ |

### 3.2 方法论提炼：四个可迁移的架构思想

**思想一：原语（Primitives）而非产品功能**

作者不按"Claude Code 有哪些按钮"来组织，而是按 **6个架构原语** 组织：

```
Subagents（子代理） → Commands（斜杠命令） → Skills（技能）
    ↓                      ↓                        ↓
Workflows（工作流编排）  Hooks（钩子拦截）     Memory（记忆系统）
```

每个原语是一个独立的概念单元，可以跨平台迁移。这对我们设计跨平台通用版有直接参考价值——不是按"WB有的""扣子有的"来分，而是按"所有AI平台都有的架构原语"来组织。

**思想二：知识地图（Meta-Knowledge）的价值**

这不是"教你怎么用 Claude Code"，而是"把整个 Claude Code 生态画成一张地图"。54k stars 不是因为内容原创性最高，而是因为**"把碎片信息整合成体系"这件事本身有巨大的时间节省价值**。

**我们可直接对标**：我们的三层记忆架构 + 知识库就是 WB+扣子生态的知识地图。

**思想三：量化社区共识**

12个工作流全部标注 star 数，技能集合标注 skill 数量。这不是简单的"列出选项"，而是**用社区投票告诉你"哪个方案经过了最大规模的验证"**。

| 工作流 | Stars | 关键特点 |
|------|:--:|------|
| Superpowers | 233k | 最完整：从头脑风暴到发版，覆盖全链路 |
| Everything Claude Code | 218k | 安全扫描+E2E测试集成 |
| Matt Pocock Skills | 136k | 强调"grill-me"挑战模式 |
| gstack | 111k | CEO/Eng/Design/Devex 多角色审查 |
| Get Shit Done | 64k | 最务实："先跑通再说" |

**我们可直接对标**：我们的第5期巡检应该引入"社区验证量化"维度——不仅我们自己评分，也参考社区的实际采用量。

**思想四："十亿美元问题"——提出比回答更重要**

项目中专门有一节"十亿美元问题"，不提供答案，只提问：

> CLAUDE.md 中该放什么？不该放什么？多久更新一次？为什么 Claude 仍忽略 CLAUDE.md 指令？

这种"以问代答"的方式，本身就是 AI 工程领域的高阶思维。**真正的高手不问"怎么用"，而问"什么时候不该用"**。

### 3.3 能力边界（局限性）

| 局限 | 说明 |
|------|------|
| 平台锁定 | 全部围绕 Claude Code，扣子/WB 用户需转换 |
| 信息密度过高 | 83条技巧没有优先级排序，新手容易迷失 |
| 更新依赖作者个人 | 单作者维护，社区贡献机制不够成熟 |
| 缺少错误反例 | 只讲"应该怎么做"，不讲"常见的错误做法" |

### 3.4 反向工程：如果我们要做一个"WB生态知识地图"

```
WB Agentic Engineering Map/
├── 核心原语（对标 claude-code-best-practice 的 Primitives）
│   ├── MEMORY（记忆系统）
│   ├── SKILLS（技能市场+自定义）
│   ├── CONNECTORS（连接器生态）
│   ├── AUTOMATIONS（自动化/日程）
│   └── EXPERTS（专家系统）
├── 工作流（对标 12 Development Workflows）
│   ├── 三方协作五步法（我们已有的）
│   ├── 双盲协作（我们已有的）
│   ├── SKILL巡检流程（我们已有的）
│   └── 内容创作全链路（公众号文章从构思到发布）

└── 十亿美元问题（对标 Billion Dollar Questions）
    ├── MEMORY.md 中该放什么？不该放什么？
    ├── 什么时候用 SKILL vs 直接用对话指令？
    └── 三层记忆的边界在哪？
```

## L4 — 内化：集成方案

### 集成方案1：升级巡检标准——引入"社区验证量化"

| 维度 | 当前做法 | 升级后 |
|------|------|------|
| 评估维度 | 我们自己五维评分 | 增加"社区采用量"维度（stars/安装量/Trending） |
| 发现方式 | 我们逐源搜索 | 增加"社区趋势监控"——每期巡检前看 Trending |
| 判断标准 | 我们认为好不好 | 社区验证 + 我们判断，双信源交验 |

```
【操作步骤】
Step 1: 每期巡检前，访问 GitHub Trending + SkillHub 热门榜
Step 2: 筛选 AI/Agent/SKILL 相关的新项目
Step 3: 评估时增加"社区验证度"评分（1-10分，基于 stars/采用量/讨论热度）
Step 4: 如果社区验证度高但我们的评估低 → 深挖原因（可能是我们漏了什么）
```

**预期效果**：巡检发现的覆盖面从"我们看到的"扩展到"社区看到的"，减少盲区。

**风险**：社区热度不等于质量（create-ex 就是反面例子——热门但不适用）。需配合我们的适用性一票否决。

### 集成方案2：构建"WB Agentic Engineering 知识地图"

| 部件 | 内容 | 复用 claude-code-best-practice 的什么 |
|------|------|------|
| **原语分类** | 按我们的核心能力（MEMORY/SKILLS/CONNECTORS/AUTOMATIONS/EXPERTS）组织 | 借鉴它的"Primitives"组织方式 |
| **工作流目录** | 列出我们所有已落地的工作流（五步法/双盲/巡检/内容创作） | 借鉴它的"Development Workflows"表格格式 |
| **SKILL生态地图** | 把我们的 SKILL 安装清单按场景分类 | 借鉴它的"Skill Collections"格式 |
| **十亿美元问题** | 列出我们面对的关键待解问题 | 直接模仿"Billion Dollar Questions"格式 |

```
【操作步骤】
Step 1: 创建 docs/WB-Agentic-Engineering-Map.md
Step 2: 按四部分组织（原语/工作流/生态/未解问题）
Step 3: 每个部分先列出已落地的，再标注缺口
Step 4: 随每次巡检更新——新发现的 SKILL 和工作流加入目录
```

**预期效果**：任何新加入的协作者（人或AI），5分钟就能了解我们的完整能力地图和当前边界。对外也是展示碳硅协同进化成果的窗口。

**风险**：维护成本。建议每次巡检后自动更新，不单独维护。

### 集成方案3：83条技巧中直接可应用的Top 10

从83条技巧中筛选对 WB 侧直接可用的 Top 10（去 Claude Code 平台绑定，提取通用原则）：

| # | 原技巧 | 通用化后 | 应用场景 |
|:--:|------|------|------|
| 1 | 始终使用 plan mode 开始 | 复杂任务先出计划再动手（← 我们已经在做的） | 任何新任务 |
| 2 | 从最小规格开始 | 不要一次性想全，先做最小可用版再迭代 | 方法论文档 |
| 3 | 上下文腐烂约 ~300-400k tokens | 主会话轮次超200或MEMORY.md超400行→开启新窗口 | 会话管理 |
| 4 | 新任务 = 新会话 | 上下文独立于任务，不交叉污染 | 并行任务 |
| 5 | 回退 > 纠正 | 走偏了直接回到上个正确状态，不要硬修 | 方案讨论 |
| 6 | PR 保持小而专注 | 每次修改范围可控，一个PR一件事 | 代码/SKILL开发 |
| 7 | 每个技能建 Gotchas 部分 | 记录踩过的坑（← 我们已有 gotchas.md） | SKILL开发 |
| 8 | "grill me on these changes" | 让AI主动质疑你的方案，不是等你说问题 | 方案评审 |
| 9 | 使用子代理进行上下文管理 | 长任务拆成子任务，每个独立上下文（← 德米斯的"标签页"思路） | 多任务并行 |
| 10 | 经常提交 | 小步快跑，不攒一大堆再提交 | 代码管理 |

```
【操作步骤】
Step 1: 将Top 10融入我们的Q1规则层（MEMORY.md）或现有方法论文档
Step 2: 每条标注来源（claude-code-best-practice）
Step 3: 下期巡检回看实际应用效果
```

**风险**：部分技巧是 Claude Code 特有的，需仔细甄别通用性。

## 总结：这个项目对我们的三层价值

```
L1 发现层：发现了"AI工程知识地图"这个品类——它不是单个工具，是工具的地图
    ↓
L3 拆解层：学到了四个可迁移思想——原语组织/知识地图/量化共识/以问代答
    ↓
L4 内化层：三个可立即启动的方案——巡检量化升级/知识地图建设/Top10技巧落地
```

**最终判断**：加入巡检资源库，定位为"知识资产类S级"——不安装、不搬运，但必须学习其组织方法和思想框架。它的价值不在内容本身，在**"怎么把散落的知识系统化"这件事的方法论**。
