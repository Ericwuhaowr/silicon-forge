# SKILL 借鉴评估框架 V1.0

> 原则：对外部的 SKILL 方法论保持学习和开放，但引入前必须通过本框架评估。适合我们的才采用，不适用的理性放弃。
> 创建：2026-06-12 | 来源：花叔 Huashu Design 2.0 / FanBox / Darwin-skill 研究实战提炼

---

## 一、五维评估标准

| 维度 | 判断问题 | 权重 | 评分方式 |
|------|---------|:---:|------|
| **适用性** | 这个机制在我们的平台上是否可实现？（花叔基于Claude Code，我们基于WorkBuddy） | 🔴 一票否决 | 不适用→直接放弃 |
| **必要性** | 不引入这个机制，我们的SKILL质量是否有明显短板？ | 高 | 1-5分 |
| **投入产出** | 引入成本（改动量+学习成本+维护成本）vs 预期质量提升，是否划算？ | 高 | 1-5分 |
| **与现有体系兼容** | 是增强现有体系还是推倒重来？（优先"增强"，慎做"替换"） | 中 | 1-5分 |
| **花叔已验证程度** | 他踩过的坑是否已固化？有无实测数据？（优先引用"有数据的"，慎跟"刚发布的"） | 中 | 1-5分 |

### 评分规则

- **一票否决**：适用性不满足 → 直接放弃，不做后续评分
- **通过线**：必要性≥3 且 投入产出≥3
- **优先级**：必要性×投入产出，乘积越高优先级越高
- **兼容性加分**："增强现有体系"优于"新建独立体系"

## 三、决策原则

1. **适用性一票否决**：平台能力不同，花叔基于Claude Code的做法不一定能在WB上实现
2. **增强优于替换**：优先选择能增强现有体系的借鉴，避免推翻重建
3. **有数据优于刚发布**：花叔已验证（有实测数据+反例）的机制优先采用，刚发布的功能等2-4周观察
4. **投入产出优先**：30分钟能改完且全局受益的项目优先级最高
5. **定期复盘**：每引入3个借鉴项后，用本框架复盘一次，校准评分标准
