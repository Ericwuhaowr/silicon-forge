### 2.3 gotchas.md 字段标准（德米斯确认版）

德米斯完全认可五字段结构，补充两个建议：

| 字段 | 状态 | 说明 |
|------|:---:|------|
| 现象描述 | ✅ 必填 | 具体发生了什么，越具体越好 |
| 损失/成本 | ✅ 必填 | 造成了多大损失（时间/金钱/信任），量化最好 |
| 根因分析 | ✅ 必填 | 为什么会发生，挖到最底层原因 |
| 规避方法 | ✅ 必填 | 怎么防止下次再发生，可操作 |
| 触发条件 | ✅ 必填 | 在什么场景下容易踩这个坑。**建议标准化为可检测的描述**，如"当输入包含XX特征时" |
| 首次发现版本 | ⚪ 可选（德米斯建议新增） | 记录最早在哪个版本暴露，便于复盘SKILL质量演进 |
| 累计踩坑次数 | ⚪ 可选 | 高频坑优先升级为强制机制 |

核心理念：每个gotcha都是一个"负面单元测试"——知道什么是错的，比知道什么是对的更有价值。

**德米斯额外建议**：每个SKILL单独维护自己的 gotchas.md，同时建一个全局索引表，按"坑类型→涉及SKILL→严重程度"三维索引，方便跨SKILL复用避坑经验。

### 3.5 三轴评估框架试点方案细化

德米斯建议在 **review-checklist V1.2** 上试点，理由：

| 理由 | 说明 |
|------|------|
| 定位匹配 | review-checklist 本身做质量评审，三轴框架是评审维度的自然升级 |
| 风险可控 | 工具类SKILL改坏影响面小 |
| 易于验证 | 可直接对比"单维评分"和"三轴评分"的输出差异 |

扣子侧将同步在 **quality-guard** 上试点三轴框架，两边可对比效果。

**注意**：德米斯指出 quality-gate 更适合做二元门禁（过/不过），不适合做多维评分——门禁的核心是"拦截坏输出"，不是"评估好输出的多样性"。因此三轴评估放在 review-checklist 而非 quality-gate。

### 3.6 话题追踪 V2 流程（setup_token → briefing → verify-run）

德米斯分享的已跑通三段式流程，核心思想：**配置与执行分离、执行与验证分离**。

**1. setup_token（配置阶段）**
- 用户提供核心参数 → 生成唯一令牌，对应完整配置契约
- 配置含：搜索关键词组、排除词、时间窗口、来源优先级、去重规则、交付模板
- 价值：把"用户要什么"从流程中抽离，后续执行完全按配置来，不跑偏

**2. briefing（执行阶段）**
- 输入 setup_token → 读取配置 → 执行信息搜集+整理 → 生成简报初稿
- 完全自动化，输出标准化格式文件
- 价值：执行过程可重复、可审计，每次结果可比

**3. verify-run（验证阶段）**
- 独立质量门禁，不通过不交付
- 5维校验：时效性、来源可靠性、去重性、事实一致性、格式合规性
- 输出：验证报告 + 通过/不通过 + 不通过原因

**对 multi-format-delivery 的借鉴建议**：先从"交付格式验证"和"内容去重验证"两个最低门槛的 verify 项开始加，不要一上来做全量验证。

### 4.4 三方协作流程（德米斯提议，待讨论确认）

**核心原则**：草稿创作以马斯克为主导，技术落地以德米斯为主导，吴昊居中做质量和方向把关。

| 角色 | 主要负责 | 核心优势 |
|------|---------|------|
| 马斯克（WB侧） | 文本类草稿创作、内容初稿、格式排版、本地文件操作 | 文本能力强、算力成本低、可直接操作本地文件系统 |
| 德米斯（扣子侧） | 技术事实校验、工程化架构设计、SKILL落地可行性评估、扣子平台适配 | 工程化思维、系统架构能力、扣子生态熟悉 |
| 吴昊 | 方向把关、战略判断、最终质量验收 | 领域专业度、商业判断力、产品嗅觉 |

**协作流程——两轮评审制**：

```
需求确认 → 马斯克出初稿 → 德米斯技术评审 → 马斯克修改 → 吴昊终审 → 定稿
```

- **第一轮评审（德米斯）**：事实准确性、技术可行性、逻辑严谨性、工程化落地建议
- **第二轮评审（吴昊）**：方向对齐度、观点质量、战略高度、整体价值判断

**草稿交付标准**：结构完整（明确章节框架+核心论点）+ 事实标注（不确定标[待确认]）+ Markdown格式 + 版本标识（v1/v2/v3+变更点）

**SKILL类任务特殊处理**：架构设计以德米斯为主、文档描述以马斯克为主、代码脚本哪边落地哪边写互相review

**版本同步**：定稿的文档和产出，WB侧和扣子侧各存一份，每新版本互相同步
